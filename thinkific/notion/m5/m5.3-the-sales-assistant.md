## How it fits together

Like any deployed agent, the assistant runs **inside `langgraph dev`** and is reached over HTTP by a chat UI. What's new here is everything the agent draws on: the **Chinook database**, its **sales playbooks**, and **mail**.

## How little of it is new code

Here is the project structure. Notice how much of the agent's **behavior** is set by plain-text files rather than by new code:

```text
sales_assistant/
├── agent.py            # assembles the agent — the only "wiring" code
├── langgraph.json      # tells langgraph dev which graph to serve
├── AGENTS.md           # operating manual — plain text, describes behavior
├── skills/             # one playbook per task — plain text, no code
│   ├── rfq-quote/
│   ├── weekly-newsletter/
│   └── territory-report/
├── subagents.py        # the four specialists
├── tools/              # tool code, each given to the agent that needs it
│   ├── sql.py          # → the chinook-analyst subagent
│   └── html.py         # → the main agent (builds HTML deliverables)
├── mail_mcp.py         # discovers the mail tools
├── mcp/                # the local mock mail server + a seed inbox
└── data/
    └── chinook.db      # the sales database (ships with the agent)
```

`AGENTS.md` and everything under `skills/` are just **natural language** — read directly by the model. The operating manual sets out who the agent works for and how to behave; each skill is a playbook for one task. For example, `AGENTS.md` states the approval rules in prose:

```text
## Approvals (human-in-the-loop)

Two actions wait for Jane to approve, edit, or reject before they take effect:

- Saving an email draft (inbox-manager): drafts are never sent, only saved.
- Adding a new customer (chinook-analyst): no row is written without her ok.
```

### The code that wires it together

The *assembly* is tiny: `agent.py` pulls the tools and subagents together and hands the result to the server in a single `create_deep_agent(...)` call.

Two key design decisions stand out in `agent.py`:

- **You hand the server a *graph*, not a `main()`.** Earlier modules ran an agent with `asyncio.run(main())`. Here you never run it yourself: `langgraph.json` points at `make_graph`, and `langgraph dev` calls it.
- **You discover the mail tools *before* building the agent.** A model can only call a tool it has been told about, and an MCP server hands over its tool list through an async call. So you `await load_mail_tools()` first, then build the agent around whatever came back.

The two human-in-the-loop approval gates (`interrupt_on`) live on the specialist subagents — not on the main agent — so they cannot be bypassed through the built-in `general-purpose` subagent, which inherits the main agent's tools.

The `langgraph.json` points at the async factory (`make_graph`) rather than a plain variable — that's how async tool discovery runs at startup:

```json
{
  "dependencies": ["."],
  "graphs": {
    "agent": "./agent.py:make_graph"
  },
  "env": "../../.env"
}
```

## Run it

```bash
cd python/m5/sales_assistant
uv run langgraph dev --allow-blocking
```

`--allow-blocking` is needed because discovering the mock mail server's tools launches a subprocess at startup — synchronous I/O that `langgraph dev`'s blocking-call detector would otherwise reject.

This starts the server on `http://localhost:2024` and opens **LangGraph Studio**. For the human-approval steps, use **agent-chat-ui**:

1. Open **[agentchat.vercel.app](https://agentchat.vercel.app)**.
2. Set **Deployment URL** to `http://localhost:2024` and **Assistant / Graph ID** to `agent`.
3. Chat as normal. When the assistant saves an email draft or adds a customer, an **approve / edit / reject** card appears right in the conversation.

### Config knobs

The assistant reads its options from the shared `python/.env`. With the standard course setup you need **nothing extra**. Optional:

- `ENABLE_SANDBOX`: add it (e.g. `ENABLE_SANDBOX=1`) to turn on the territory chart.
- `TAVILY_API_KEY`: set during course setup; enables the newsletter's web research.

## What it can do

With the assistant running, try these three task sequences:

1. **Process an incoming RFQ**: `Check the inbox for any quote requests and process them.`
2. **Write the weekly newsletter**: `Write this week's "This Week in Music" newsletter.` *(requires `TAVILY_API_KEY`)*
3. **Report on the territory**: `How's my book of business looking? Give me a territory report.` *(add `ENABLE_SANDBOX=1` to include a chart)*

## Recap

- The **Chinook Sales Assistant** runs inside `langgraph dev` and draws on the **Chinook database**, its **sales playbooks**, and **mail**.
- Much of the agent's behavior lives in **plain text**: `AGENTS.md` and the `skills/` playbooks, written in natural language. The code is small and focused (tools and subagents), wired together by a single `create_deep_agent(...)` call.
- The agent is served as a **graph** (a `make_graph` factory), not run with `main()`, and it **discovers its MCP tools up front**.
- Run it with `uv run langgraph dev --allow-blocking` and chat via **agent-chat-ui** (agentchat.vercel.app); the **two approval steps** (saving a draft, adding a customer) pause and render as inline approve / edit / reject cards.
