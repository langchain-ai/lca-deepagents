The Sales Assistant

🎥 Video Walk-through [click to expand]


In Lesson 1 you saw what the Chinook Sales Assistant is and which Deep Agents feature each part uses. In Lesson 2 you learned how to serve an agent with langgraph dev. Now we put it together: meet the finished assistant, see how little new code you write to assemble it, and run it locally.

The assistant works for Jane Peacock, a Sales Support Agent at Chinook (an online music distributor). "Her book of business" means the customers whose support rep is Jane — Customer.SupportRepId = 3 in the database. The assistant helps her answer quote requests, keep customer records current, research the market, and report on her territory.

Note: This lesson sets the assistant up and runs it. The three task sequences it can perform are demonstrated in the video walkthrough above.

How it fits together

Like any deployed agent, the assistant runs inside langgraph dev and is reached over HTTP by a chat UI. What's new here is everything the agent draws on: the Chinook database, its sales playbooks, and Gmail.

How little of it is new code

Here is the project. There is real code here — the tools, the subagents, a mock mail server. But notice how much of the agent's behavior is set by plain-text files rather than by new code you write:

sales_assistant/
├── agent.py            # assembles the agent — the only "wiring" code
├── langgraph.json      # tells langgraph dev which graph to serve
├── AGENTS.md           # operating manual — plain text, describes behavior
├── skills/             # one playbook per task — plain text, no code
│   ├── rfq-quote/
│   ├── weekly-newsletter/
│   └── territory-report/
├── subagents.py        # the four specialists
├── tools/              # tool code, each given to the agent that needs it
│   ├── sql.py          # → the chinook-analyst subagent
│   └── html.py         # → the main agent (builds HTML deliverables)
├── gmail_mcp.py        # discovers the Gmail tools (mock or real)
├── mcp/                # the local mock Gmail server + a seed inbox
└── data/
    └── chinook.db      # the sales database (ships with the agent)

AGENTS.md and everything under skills/ are just natural language — read directly by the model, in any language it understands. The operating manual sets out who the agent works for and how to behave; each skill is a playbook for one task. For example, AGENTS.md states the approval rules in prose:

## Approvals (human-in-the-loop)

Two actions wait for Jane to approve, edit, or reject before they take effect:

- Saving an email draft (inbox-manager) — drafts are never sent, only saved.
- Adding a new customer (chinook-analyst) — no row is written without her ok.

That is Lesson 1's point: much of the work is describing behavior, not writing new code.

The code that wires it together

You still write code — your tools (tools/) and the specialists (subagents.py). But the assembly is tiny: agent.py pulls those pieces together and hands the result to the server in a single create_deep_agent(...) call.

Here is the file in full. It opens at make_graph, the factory that assembles the agent — scroll up for the imports and system prompt, down for the rest:

# python/m5/sales_assistant/agent.py
"""The Chinook Sales Assistant — served as a graph for `langgraph dev`.

This module is just the agent. The Gmail MCP plumbing lives in `gmail_mcp.py`;
all this file does is assemble the agent around whatever tools that discovery
returns.

`langgraph.json` points at the async factory `make_graph`, which langgraph awaits
once at startup. We use a factory (not a plain module-level `agent = ...`) for
one reason: the Gmail tools come from an MCP server, and asking an MCP server for
its tool list is an async call — so we discover the tools up front, then build
the agent around whatever we found. If Gmail is unreachable, discovery returns no
tools and the assistant simply comes up without them.
"""

from __future__ import annotations

import logging
import os
import sys
from pathlib import Path

# Make the shared workshop model module importable (python/models.py). models.py
# also calls load_dotenv, so env vars (API keys, GMAIL_BACKEND) are loaded.
# parents[2] = python/ (this file is python/m5/sales_assistant/agent.py).
sys.path.insert(0, str(Path(__file__).resolve().parents[2]))
from deepagents import create_deep_agent  # noqa: E402
from deepagents.backends import FilesystemBackend  # noqa: E402
from gmail_mcp import load_gmail_tools  # noqa: E402
from langchain_quickjs import CodeInterpreterMiddleware  # noqa: E402
from subagents import build_subagents  # noqa: E402
from tools.html import markdown_to_html  # noqa: E402

from models import strong_model  # noqa: E402

logger = logging.getLogger("chinook-sales-assistant")

HERE = Path(__file__).resolve().parent


SYSTEM_PROMPT = (
    "You are a sales assistant for Jane Peacock, a Sales Support Agent at "
    "Chinook, an online music distributor. Follow your operating manual (loaded "
    "from your memory) and use the matching playbook from /skills/ for each task."
)


async def make_graph():
    """Async graph factory — built once by `langgraph dev` at startup."""
    # Real local disk under this module only (virtual_mode locks the agent to
    # this directory: skills/, AGENTS.md, outputs/). Suitable for local dev;
    # not for a multi-tenant web server.
    backend = FilesystemBackend(root_dir=str(HERE), virtual_mode=True)

    enable_search = bool(os.environ.get("TAVILY_API_KEY"))
    if not enable_search:
        logger.info("TAVILY_API_KEY not set — newsletter research subagent disabled.")

    gmail_tools = await load_gmail_tools()

    # The main agent holds NO gated tools. Gmail (gmail_create_draft) lives on
    # the inbox-manager subagent and SQL writes (add_customer) on chinook-analyst,
    # each with its own approval gate — so the always-present general-purpose
    # subagent, which inherits the main agent's tools, can never reach a write
    # ungated. The main agent keeps only safe tools plus the interpreter.
    tools = [markdown_to_html]

    # Optional sandbox-backed chart tool (the course's optional feature).
    if os.environ.get("ENABLE_SANDBOX"):
        from tools.chart import render_chart

        tools.append(render_chart)

    return create_deep_agent(
        model=strong_model,
        tools=tools,
        system_prompt=SYSTEM_PROMPT,
        subagents=build_subagents(
            backend, enable_search=enable_search, gmail_tools=gmail_tools
        ),
        skills=["/skills"],
        memory=["/AGENTS.md"],
        backend=backend,
        middleware=[CodeInterpreterMiddleware()],
        name="chinook-sales-assistant",
    )

Two things are worth pausing on:

  • We hand the server a graph, not a main(). Earlier modules ran an agent with asyncio.run(main()). Here we never run it ourselves — langgraph.json points at make_graph, and langgraph dev calls it. We expose the agent; the server runs it.
  • We discover the Gmail tools before building the agent. A model can only call a tool it has been told about, and an MCP server hands over its tool list through an async call. So we await load_gmail_tools() first, then build the agent around whatever came back. If the Gmail server is unreachable, discovery simply returns nothing and the assistant comes up without email — one bad server never stops it.

And the langgraph.json that points at that factory (python/m5/sales_assistant/langgraph.json):

{
  "dependencies": ["."],
  "graphs": {
    "agent": "./agent.py:make_graph"
  },
  "env": "../../.env"
}

Compare this to Lesson 2's lab, where the graph was a plain variable ("./agent.py:graph"). Here it points at a function (make_graph) that the server awaits — that's how we get to run async tool discovery at startup.

Run it

Serve the assistant the same way you served the Lesson 2 agent — with one extra flag:

cd python/m5/sales_assistant
uv run langgraph dev --allow-blocking

Why --allow-blocking? Discovering the mock Gmail server's tools launches a subprocess at startup — synchronous I/O that langgraph dev's blocking-call detector would otherwise reject.

This starts the server on http://localhost:2024 and opens LangGraph Studio, where you can chat with the assistant and step through its graph — exactly as in Lesson 2. When the assistant goes to save an email draft or add a new customer, the run pauses for you to approve, edit, or reject before it continues.

Studio is all you need for this lab. If you would like a cleaner, end-user-style chat — where those approvals render as inline cards — there is an optional hosted UI:

Optional: a richer chat UI (hosted agent-chat-ui)


LangChain hosts a ready-made chat interface called agent-chat-ui — no install required:

  1. Open the hosted UI at agentchat.vercel.app.
  2. Point it at your running server: Deployment URL http://localhost:2024, Assistant / Graph ID agent.
  3. Chat as normal. When the assistant saves an email draft or adds a customer, an approve / edit / reject card appears right in the conversation.

You can also run it yourself with Node and pnpm from the agent-chat-ui repo, but the hosted version needs no setup.

Optional config knobs

The assistant reads its options from the shared python/.env. With the standard course setup you need nothing extra — Gmail uses a local mock and the sandbox is off. The optional knobs:

  • GMAIL_BACKEND — mock (the default; offline, zero setup) or real (OAuth Gmail). Both expose the same tool surface, so the agent code never changes.
  • ENABLE_SANDBOX — add it (e.g. ENABLE_SANDBOX=1) to turn on the territory chart. Leave it out to run without — you still get a numeric report.
  • TAVILY_API_KEY — set during course setup; enables the newsletter's web research.

What it can do

With the assistant running, you can ask it to handle a sales rep's day-to-day work. The video walks through three task sequences, each showing off different Deep Agents features:

  1. Process an incoming RFQ — read a quote request from the inbox, look up the customer and prices, compute the quote exactly with the interpreter, have a reviewer subagent check it, draft the reply for approval, and log it.
  2. Write the weekly newsletter — fan out parallel research subagents (one per genre), then assemble an HTML newsletter on disk.
  3. Report on the territory — pull metrics from the database and, with the sandbox enabled, render a chart.

Try them yourself once the server is up.

Check your understanding

Question: Why does agent.py use an async make_graph factory instead of a module-level agent = create_deep_agent(...)?
A) To make the agent start faster
B) So the agent can be invoked with asyncio.run()
C) Because the Gmail tools come from an MCP server, and getting its tool list is an async call that must finish before the agent is built
D) Because langgraph dev cannot load module-level variables
Answer: C ✅
A model can only call tools it has been told about. The Gmail tools come from an MCP server whose tool list arrives via an async call, so discovery must be awaited before the agent is assembled — hence an async factory.
Question: What is the --allow-blocking flag on langgraph dev for?
A) Discovering the MCP server tools does a little synchronous I/O at startup, which the blocking-call detector would otherwise flag
B) It allows the agent to block on human approvals
C) It enables blocking network calls to Gmail
D) It is required for every langgraph dev server
Answer: A ✅
Launching the mock Gmail server to list its tools is synchronous I/O at startup. langgraph dev normally raises on blocking calls; the flag marks this expected one as allowed.
Question: The two human approvals (saving an email draft, adding a customer) are gated on specialist subagents rather than the main agent. Why?
A) To make the approval cards look nicer
B) Subagents run faster than the main agent
C) It is the only place interrupts are allowed
D) The general-purpose subagent inherits the main agent tools, so a gated tool on the main agent could be reached ungated; keeping writes on gated specialists makes the design fail-closed
Answer: D ✅
The always-present general-purpose subagent inherits the main agent's tools. If a write lived on the main agent it could be reached ungated through delegation. Placing both writes on gated specialists means the only path to either write runs through its approval gate.
Question: In this project, what are AGENTS.md and the files under skills/?
A) Compiled configuration generated at build time
B) Plain-text files that describe the agent behavior — an operating manual and one playbook per task
C) Python modules that define the subagents
D) Database fixtures for the Chinook data
Answer: B ✅
AGENTS.md is the operating manual and each skill is a step-by-step playbook — all natural language describing what the agent should do, not code.
