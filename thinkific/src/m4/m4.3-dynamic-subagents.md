## From one subagent to many

Delegation meant handing **one task** to **one subagent** and getting one result back: a `genre-researcher` for Rock, another call for Metal, a third for Latin. You wrote the `task` calls yourself, one per genre, because there were only four genres.

Now say the job is 1,000 sales transcripts, same task on every one: summarize it. There's no longer a short list the model can enumerate in its head, and someone still has to track which ones are done.

<p style="margin: 0;">Leave that to the model and you've put the exact thing that breaks agents at scale (tracking state across a shrinking context window) back in its head instead of off of it.</p>

<details>
<summary>What happens if you leave this to the model?</summary>

In practice, a model asked to take on all 1,000 transcripts itself wouldn't just cleanly fail. Depending on the model, it would more likely:

- Refuse outright, saying it can't process that many in one pass.
- Try to chunk the work into rough ranges across subagent calls, e.g. transcripts 0-100, then 100-200, and so on, hoping it remembers where it left off.
- Try to fire off everything in batches of roughly 20 calls at a time, since that's close to the practical ceiling on how many tool calls a model can issue in parallel.

None of these guarantee every transcript actually gets touched; each still depends on the model, not code, getting the bookkeeping right.

</details>

### Programmatic orchestration

Dynamic subagents solve this with [programmatic orchestration](https://www.langchain.com/blog/introducing-dynamic-subagents-in-deep-agents). Instead of issuing subagent tasks through tool calls turn by turn, the model writes a short script that orchestrates and calls subagents. It leans on code patterns it already knows: looping, branching, concurrency. That script runs in a lightweight interpreter.

Here's that same difference written as code, since that's really what's changing:

```python
# the model enumerates each item itself, and has to remember its own place
task(description="summarize transcript 1 of 1,000")
task(description="summarize transcript 2 of 1,000")
# ...however many calls it takes, nothing guarantees it reaches transcript 1,000...
```

```python
# written once, run by code: the loop is the guarantee, not the model's memory
for transcript in transcripts:
    task(description=f"summarize {transcript}")
```

`task()` itself didn't change; it's the same subagent call either way. What changed is *where the decision to keep going lives*:

- **First version:** in the model, however many calls it takes.
- **Second version:** in a loop the interpreter runs once.

**Dynamic subagents** is Deep Agents' name for that programmatic orchestration: subagents dispatched by code, not by the model, one turn at a time.

That loop doesn't have to be a literal `for`: it could be a map over a list, a parallel fan-out, or a branching pipeline. The only thing that matters is that it's a fixed shape written in code, not something the model has to reconstruct from memory on every turn.

## The interpreter can call `task()` too

The [code interpreter](../m2/m2.4-interpreter.html) exposes a global `task()` function that bridges straight to the exact same `task` tool a main agent would call directly.

Concretely, in Deep Agents:

- `task` takes a `description` and a `subagent_type`.
- When the interpreter (`CodeInterpreterMiddleware`) has subagents configured, it freezes a `task()` function into its global scope. Calling it from JavaScript inside the interpreter invokes that same underlying tool.
- Each call still runs a fresh, isolated invocation of that subagent: a clean state with no memory of prior runs, executed end to end. Nothing about *what* gets invoked changes based on *who* called `task`, the model directly or a script running in the interpreter.

The same idea applies to ordinary tools, not just subagents: interpreter code can call `tools.*` directly (e.g. `tools.glob(...)`) instead of the model invoking each tool one at a time. LangChain calls this **programmatic tool calling (PTC)**; `task()` is the subagent-side equivalent. PTC is off by default and needs an explicit allowlist on the interpreter middleware.

  dots.forEach(function (d, i) { d.addEventListener('click', function () { current = i; render(); }); });
  stage.addEventListener('click', advance);
  stage.addEventListener('keydown', function (e) {
    if (e.key === 'ArrowRight' || e.key === ' ') { e.preventDefault(); advance(); }
    if (e.key === 'ArrowLeft')                   { e.preventDefault(); retreat(); }
  });
  var tx = 0;
  stage.addEventListener('touchstart', function (e) { tx = e.touches[0].clientX; }, { passive: true });
  stage.addEventListener('touchend', function (e) {
    var dx = e.changedTouches[0].clientX - tx;
    if (Math.abs(dx) > 40) dx < 0 ? advance() : retreat();
  });
})();
</script>

<Tip>

**Governance note.** [`interrupt_on`](../m1/m1.8-hitl.html) pauses a tool call for human approval before it executes, but only for tool calls the model makes directly. `task()` calls made from inside an already-running `eval` call don't go through that path, so a per-tool `interrupt_on` gate on the parent agent's `task` tool won't catch them.

If you need a human in the loop before orchestration runs, gate the `eval` tool itself, not the individual subagent dispatches inside it. `interrupt_on` configured on an individual subagent's own spec still applies, since that gate runs inside the subagent's own invocation, not the parent's dispatch.

</Tip>

## Why write code instead of delegating turn-by-turn

Agents have trouble doing work reliably at scale, for two structural reasons:

1. Context windows are finite.
2. The usual fix, summarization, is lossy.

Dynamic subagents support two things that turn-by-turn delegation doesn't guarantee:

- **Deterministic coverage at scale.** A model deciding turn by turn whether it's covered everything can quietly stop at 75 of 500 items and call it done; a `for batch in batches` loop doesn't have that option, it touches every batch.
    - On a long-context benchmark called OOLONG, a plain agent at 128k tokens didn't just answer wrong; it often gave up outright, reporting it couldn't compute the result. An interpreter-enabled agent on the same benchmark did dramatically better (numbers under "Recursive language models" below).
- **Reliable complex orchestration.** Writing orchestration as code is more reliable than having the model reproduce it as a sequence of tool calls, especially for fan-out-and-synthesize work, multi-phase pipelines, or conditional branching: the pipeline can branch, run in parallel, or run sequentially, shaped however the task needs, instead of being constrained to one tool call per model turn.

## How the model picks a pattern

Only "workflow" is configured. In a prompt, "workflow" flips on a single instruction: write dispatch code instead of delegating one call at a time.

Everything past that point is the model writing ordinary code. Whether it ends up branching by category, fanning out in parallel, looping until nothing new turns up, or running an elimination bracket depends on how the model reasons about the shape of the task while it writes the script.

Mixed-type inputs tend to produce a branch. A pile of independent, uniform items tends to produce a parallel map. An open-ended search tends to produce a loop. The six shapes below are common outcomes of that reasoning, not configuration options you set anywhere.

## Recursive language models (RLMs)

Deep Agents implements a documented technique called the **recursive language model (RLM)**, introduced by Alex Zhang, Tim Kraska, and Omar Khattab at MIT CSAIL ([paper](https://arxiv.org/abs/2512.24601)).

> **Core insight:** instead of a model treating a long context as one monolithic input, split it into pieces, call each piece, and combine the results, recursing on any piece still too big to handle directly.

Deep Agents' dynamic subagents use that same divide-and-conquer technique, but dispatch full agents rather than recursing on the same model call. The LangChain team's more precise term for that is **recursive agents (RA)**; "RLM" is the popular shorthand you'll see used for the pattern generally.

The paper reports RLMs handling inputs "up to two orders of magnitude beyond a model's context window," with median gains on GPT-5 ranging from 13% (vs. Claude Code) to 130% (vs. CodeAct with sub-calls) across four long-context tasks.

### Our own testing

The OOLONG benchmark (long-context reasoning and aggregation) is the same shape of test the paper runs, and running it ourselves against the interpreter-enabled agent from this lesson agrees with the paper's direction, at more than one context length:

| Context length | Without interpreter | With interpreter |
|---|---|---|
| 64k tokens | 0.58 | 0.67 |
| 128k tokens | 0.44 | 0.79 |

Same caveat as the paper's own framing: an untuned smoke test, not a finished benchmark, but the gap agrees with the direction the paper reports. ([source](https://www.langchain.com/blog/how-to-use-rlms-in-deep-agents))

## When to reach for this vs. a plain subagent team

<Tip>

**Reach for dynamic subagents when:**
- The number of units of work is large, fixed, or only known at runtime (pages, rows, files, transcripts)
- You need a **guarantee** that every unit gets covered, not a best-effort pass
- The orchestration itself is non-trivial: branching, parallel fan-out, or a pipeline shape the model would otherwise have to reconstruct every turn

**Stick with an ordinary subagent team when:**
- There's a small, fixed number of specialized roles (a researcher per genre, a reviewer, an analyst)
- You're delegating *by role*, not fanning out over a large, uniform collection of items

</Tip>

## Recap

- **Dynamic subagents** means subagents dispatched from a script running in a **[code interpreter](../m2/m2.4-interpreter.html)**, instead of the model calling `task` turn by turn.
- The interpreter's `task()` isn't a separate mechanism: it's the exact same `task` tool a main agent calls directly, so every dispatch is still a full agent with its own tools and state, whether the model calls it or a script does.
- The word **"workflow"** in a prompt is what triggers code-based dispatch; which orchestration *shape* the model then writes (classify-and-act, fan-out, loop, tournament...) isn't chosen from a menu, it emerges from how the model reasons about the task while writing the script.
- The payoff is **deterministic coverage** and **reliable complex orchestration**, both hard to get from a model deciding turn by turn.
- `task()` dispatched from inside `eval` bypasses per-dispatch `interrupt_on` gating; gate the `eval` tool itself if you need a human in the loop.
- These patterns implement a documented technique, the **recursive language model (RLM)**: split a long context into pieces, call each, combine, recurse on anything still too big. LangChain's more precise term for the Deep Agents version is **recursive agents (RA)**; the RLM paper reports GPT-5 median gains ranging from 13% to 130% over different baselines on long-context tasks.
- Reach for dynamic subagents when the number of work units is large or open-ended and you need guaranteed coverage; stick with a plain subagent team for a small, fixed set of specialist roles.
