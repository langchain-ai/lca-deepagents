## Not the same "async" you might expect

Async subagents let a supervisor agent launch background tasks that return immediately, so the supervisor can continue interacting with the user while subagents work concurrently. The supervisor can check progress, send follow-up instructions, or cancel tasks at any point.

It's tempting to picture an async subagent as a sync subagent that just happens to run on a background thread while the main agent keeps talking to the user. However, that's not the model Deep Agents uses.

- A synchronous subagent (module 4.1, 4.2) is invoked through the built-in `task` tool, which calls `create_agent` **in the same process**, waits for it to finish, and returns the result.

- An async subagent is invoked through a different set of tools that go over the **LangGraph SDK** to a graph running on a **remote [Agent Protocol](https://github.com/langchain-ai/agent-protocol) server** (either a managed LangGraph Platform deployment or a self-hosted one). The main agent doesn't wait for that run to finish. It gets a task ID back right away and can check on it whenever it wants.

That "remote server" isn't exotic infrastructure. It's the same kind of deployment you'll set up yourself in Module 5, when you run `langgraph dev` to serve an agent over HTTP. An async subagent target is just another deployed graph, reachable the same way.

## The five tools

Deep Agents adds an `AsyncSubAgentMiddleware` that exposes five tools to the main agent, instead of the single `task` tool synchronous subagents use:

| Tool | What it does |
|---|---|
| `start_async_task` | Launches a background run on the named async subagent. Returns a `task_id` immediately. |
| `check_async_task` | Looks up the current status of a task by `task_id`. If it's finished, the result comes back too. |
| `update_async_task` | Sends new instructions to a still-running task, on the same thread. |
| `cancel_async_task` | Stops a running task. |
| `list_async_tasks` | Lists every tracked task and its live status, optionally filtered (`running`, `success`, `error`, `cancelled`). |

`start_async_task` doesn't block. It creates a thread on the remote server, kicks off a run on it, and hands the model a `task_id` right away, its own model call continues without waiting on the subagent at all.

## Setup

Configuring an async subagent looks like configuring a sync one, but instead of a system prompt and tools, you point at a deployed graph:

```python
from deepagents import create_deep_agent
from deepagents.middleware.async_subagents import AsyncSubAgentMiddleware

async_middleware = AsyncSubAgentMiddleware(
    async_subagents=[
        {
            "name": "researcher",
            "description": "Long-running research agent for deep, multi-source analysis.",
            "graph_id": "research_agent",
            "url": "https://my-deployment.langsmith.dev",
        }
    ],
)

agent = create_deep_agent(
    model="openai:gpt-5.5",
    middleware=[async_middleware],
)
```

- **`name`**: what the main agent calls this subagent type when it calls `start_async_task`.
- **`description`**: how the main agent decides when to delegate to it, same role as a sync subagent's description.
- **`graph_id`**: the graph or assistant ID on the remote server.
- **`url`**: where that server lives. Omit it to connect to a local graph over ASGI transport instead of HTTP; for LangGraph Platform, authentication is handled automatically from your environment (`LANGGRAPH_API_KEY`, `LANGSMITH_API_KEY`, or `LANGCHAIN_API_KEY`).

You can register more than one async subagent type, exactly like the `subagents` list for sync subagents, each with its own `graph_id` and optionally its own `url`.

## Tracking a task

Every launched task is tracked as an `AsyncTask` and persisted in the main agent's state, under an `async_tasks` key: a `task_id` (the thread ID), which agent type launched it, its `run_id`, its `status`, and when it was created, last checked, and last updated.

That matters for one specific reason: **it survives context compaction.** If the conversation gets long enough that older messages are summarized away, the task ID doesn't disappear with them, the agent can still call `list_async_tasks` to recall what's out there and check on it.

---

## How the main agent is expected to behave

Launching a task and immediately checking on it defeats the point, so the middleware's own instructions to the model are strict about the workflow:

<Tip>

- **Launch, then stop.** After `start_async_task`, report the task ID and return control. Don't immediately call `check_async_task` on it.
- **Check only on request.** Call `check_async_task` when the user asks for a status update, not on a timer or in a loop.
- **Never trust a status already in the transcript.** A task that shows `"running"` earlier in the conversation may be done by now. Always call a tool to get the current status rather than repeating a stale one.
- **`update_async_task` isn't a message to a listener.** It interrupts the current run and starts a fresh one on the same thread, so the subagent sees the full history plus the new instructions. The `task_id` doesn't change.

</Tip>

## Sync vs. async, side by side

| | Synchronous subagent | Async subagent |
|---|---|---|
| Invoked via | built-in `task` tool | `start_async_task` + friends |
| Runs where | in-process, via `create_agent` | remote server, via the LangGraph SDK |
| Main agent | blocks until it returns | continues immediately |
| Identified by | nothing, it's just a tool call | `task_id` (persisted in state) |
| Good fit for | focused work the main agent needs before it can continue | long-running or concurrent work the main agent shouldn't wait on |

<Tip>

**Reach for an async subagent when:**
- The work genuinely takes a while, longer than you want the main agent sitting idle
- You want several long jobs running concurrently while the user keeps interacting with the main agent
- The work is better run on a separate, specialized deployment than in-process

**Stick with a synchronous subagent when:**
- The main agent needs the result before it can take its next step
- The task is quick, and the SDK round-trip and remote deployment aren't worth it

</Tip>

---

## Recap

- An async subagent isn't a background thread. It's a separate agent running on a **remote Agent Protocol server**, reached through the **LangGraph SDK**.
- Five tools replace the single `task` tool: `start_async_task`, `check_async_task`, `update_async_task`, `cancel_async_task`, `list_async_tasks`.
- `start_async_task` returns a `task_id` immediately; the main agent is expected to report it and move on, not poll.
- Tasks are tracked as `AsyncTask` records in state under `async_tasks`, which is why they survive context compaction.
- Any status already in the conversation may be stale — checking always means calling a tool, never repeating what was said earlier.
- Configuring an async subagent means pointing at a deployed graph (`graph_id`, `url`), the same kind of deployment you'll build yourself in Module 5.

(function () {
  var tabs = document.querySelectorAll('.lt-tab');
  function show(p) {
    tabs.forEach(function (t) {
      var on = t.getAttribute('data-p') === p;
      t.classList.toggle('active', on);
      t.setAttribute('aria-selected', on ? 'true' : 'false');
    });
    document.querySelectorAll('.lt-panel').forEach(function (panel) {
      panel.classList.toggle('active', panel.id === 'p-' + p);
    });
  }
  tabs.forEach(function (t) {
    t.addEventListener('click', function () { show(t.getAttribute('data-p')); });
  });
})();
