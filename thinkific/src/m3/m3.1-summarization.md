**Script:** This lesson covers how Deep Agents handles long conversations automatically. As you chat with an agent across many turns, messages accumulate and the token count rises. Left unchecked, it would eventually hit the model's context limit. SummarizationMiddleware deals with this automatically by watching the context fill up and compacting old messages into a summary before they ever reach the model.

[Show diagram]

Think of the context window like a thermometer. Every message adds tokens and the level rises. When it crosses 85% of the model's limit, the middleware fires. At that point, it picks a cutoff point, keeps the most recent messages intact, and summarizes everything older into a single block. The original messages are replaced in the model's view by that summary, so the token count drops sharply and the conversation can continue.

This scales. On a second compaction event, the prior summary is treated just like any other old message, so it gets folded into a new combined summary along with whatever else has aged past the cutoff. Every compaction event produces exactly one summary block, no matter how many rounds have happened before.

[Scroll to "What the summary contains"]

So what actually goes into the summary? The middleware makes a separate LLM call, using the same model as the agent, with a structured prompt that extracts four things [highlight as I go]:

The session intent, meaning what the user is trying to accomplish. Key facts and decisions from the conversation. Any artifacts like files that were created or modified. And what still remains to be done.

That resulting text is then injected as a single message at the top of the preserved recent messages, so the model picks up as if it had been there the whole time.

[Scroll to "Seeing it in action" and highlight the model.profile line]

The default threshold is 85% of the model's real context window, which for Claude Haiku 4.5 is 170,000 tokens. To actually see it fire in a demo, we override model.profile["max_input_tokens"] to a small value so the threshold is hit after just a few turns.

As agents grow more capable, tasks span more steps, sessions run longer, and tool results pile up. Token counts climb steadily toward the model's limit, and left unchecked, a long-running agent eventually stalls.

Deep Agents handles this automatically with **SummarizationMiddleware**; it watches the context fill up and compacts old messages into a summary before the model ever sees them.

Think of the context window like a thermometer. Every new message adds tokens and the level rises. When it crosses **85%** of the model's limit, the middleware fires.

## What the summary contains

The middleware makes a separate LLM call to generate the summary. It uses a structured prompt that asks the model to extract:

- **Session intent**: what the user is trying to accomplish
- **Key facts & decisions**: important context from the conversation
- **Artifacts**: files or resources created or modified
- **Next steps**: what remains to do

The resulting text is injected as a single `HumanMessage` at the top of the preserved recent messages. The model picks up the conversation as if it had been there the whole time.

---

## Seeing it in action

The default trigger is 85% of the model's real context window: 170,000 tokens for Claude Haiku 4.5. To observe it in a demo we set `model.profile["max_input_tokens"]` to a small value so the threshold is hit after a few turns:

```python
# python/m3/m3.1_summarization.py
from models import model

# Lower the reported context window so summarization fires at ~595 tokens
# (85% of 700) instead of the real threshold. Must pass model object, not string.
model.profile = {**model.profile, "max_input_tokens": 700}

agent = create_deep_agent(
    model=model,
    checkpointer=MemorySaver(),
    system_prompt="You are a helpful assistant. Keep every response to one sentence.",
)
```

```bash
cd python
uv run python m3/m3.1_summarization.py
```

## Tool result offloading

Summarization deals with messages that accumulate over many turns. A different problem is a single tool call that returns an enormous result - a full database export, hundreds of search matches, or a long code generation output. Storing the full result inline would spike the token count in one shot.

<br>

**Script:** Summarization deals with messages that build up over many turns. But there can be a different problem: a single tool call that returns an enormous result, such as a full database export, hundreds of search matches, or a long file. That would spike the token count in just one shot.

[Scroll to "Tool result offloading"]

When a tool result is too large (over roughly 20,000 tokens) FilesystemMiddleware automatically offloads it to the filesystem and replaces it with a short preview and a file path the agent can read on demand.

[Highlight the example message block]

So instead of the full result, the model sees this. First, it's told the result was too large and where it was saved on the filesystem. Then it's told it can use read_file to retrieve it, but to only read part at a time. And finally it gets a preview: the first 5 lines and the last 5 lines, so it has enough context to know what the result contains before deciding how to proceed.

[Scroll to the exclusions paragraph]

The filesystem tools themselves are excluded from this, so things like read_file, grep, and ls. They either have their own truncation built in, or offloading them wouldn't make sense.

In short: summarization handles gradual accumulation across turns, and tool result offloading handles single large spikes. Together they're what allows an agent to run indefinitely without hitting context limits.

</details>

**FilesystemMiddleware** handles this automatically with tool result offloading. When a tool returns a result that exceeds roughly **20,000 tokens**, the middleware:

1. Writes the full result to `/large_tool_results/<tool_call_id>` on the agent's filesystem
2. Replaces the inline `ToolMessage` with a short preview (first and last 5 lines) and a file reference

The message the model actually sees looks like this:

```
Tool result too large, the result of this tool call <id> was saved in the
filesystem at this path: /large_tool_results/<tool_call_id>

You can read the result from the filesystem by using the read_file tool,
but make sure to only read part of the result at a time.

Here is a preview showing the head and tail of the result:

     1  | SELECT id, name, email FROM users WHERE ...
     2  | 1, Alice, alice@example.com
     3  | 2, Bob, bob@example.com
     4  | 3, Carol, carol@example.com
     5  | 4, Dave, dave@example.com
... [1 823 lines truncated] ...
  1828  | 1827, Zara, zara@example.com
  1829  | 1828, Zoe, zoe@example.com
  1830  | 1829, ...
```

The agent can then call `read_file` with `offset` and `limit` to page through the full result on demand - reading only the portion that is relevant rather than loading everything into context at once.

The tools `ls`, `glob`, `grep`, `read_file`, `edit_file`, and `write_file` are excluded from offloading. They either have their own built-in truncation, return minimal confirmation text, or (for `read_file`) re-reading a truncated file would not help.

## Recap

- **SummarizationMiddleware** intercepts every LLM request via `wrap_model_call`; it runs before the model sees any messages
- When total tokens exceed 85% of `max_input_tokens`, old messages are partitioned at the **keep cutoff** and replaced with a summary
- The summary is generated by a separate LLM call using the same model as the agent
- `state["messages"]` is never mutated; summarization is tracked in a private `_summarization_event` field
- Passing `model.profile["max_input_tokens"]` overrides the threshold, making it practical to demo
- **FilesystemMiddleware** also offloads individual tool results that exceed ~20k tokens, writing them to `/large_tool_results/` and replacing them with a preview + file reference
