When a conversation runs long enough, older messages push the total token count toward the model's context limit. Deep Agents handles this automatically with **SummarizationMiddleware**; it watches the context fill up and compacts old messages into a summary before the model ever sees them.

---

## How the context window fills up

Think of the context window like a thermometer. Every new message adds tokens and the level rises. When it crosses **85%** of the model's limit, the middleware fires.

## What the summary contains

The middleware makes a separate LLM call to generate the summary. It uses a structured prompt that asks the model to extract:

- **Session intent**: what the user is trying to accomplish
- **Key facts & decisions**: important context from the conversation
- **Artifacts**: files or resources created or modified
- **Next steps**: what remains to do

The resulting text is injected as a single `HumanMessage` at the top of the preserved recent messages. The model picks up the conversation as if it had been there the whole time.

---

## Seeing it in action

The default trigger is 85% of the model's real context window: 170,000 tokens for Claude Haiku 4.5. To observe it in a demo we set `model.profile["max_input_tokens"]` to a small value so the threshold is hit after a few turns:

<Tabs>
<Tab title="Python">

```python
# python/m3/m3.x_summarization.py
from models import model

# Lower the reported context window so summarization fires at ~595 tokens
# (85% of 700) instead of the real threshold. Must pass model object, not string.
model.profile = {**model.profile, "max_input_tokens": 700}

agent = create_deep_agent(
    model=model,
    checkpointer=MemorySaver(),
    system_prompt="You are a helpful assistant. Keep every response to one sentence.",
)
```

When summarization fires, a private `_summarization_event` key appears in the agent's state. The `state["messages"]` list itself is never modified; the summarization happens only on what gets *sent* to the model each turn.

## Tool result offloading

Summarization deals with messages that accumulate over many turns. A different problem is a single tool call that returns an enormous result - a full database export, hundreds of search matches, or a long code generation output. Storing the full result inline would spike the token count in one shot.

**FilesystemMiddleware** handles this automatically with tool result eviction. When a tool returns a result that exceeds roughly **20,000 tokens**, the middleware:

1. Writes the full result to `/large_tool_results/<tool_call_id>` on the agent's filesystem
2. Replaces the inline `ToolMessage` with a short preview (first and last 5 lines) and a file reference

The message the model actually sees looks like this:

```
Tool result too large, the result of this tool call <id> was saved in the
filesystem at this path: /large_tool_results/<tool_call_id>

You can read the result from the filesystem by using the read_file tool,
but make sure to only read part of the result at a time.

Here is a preview showing the head and tail of the result:

     1  | SELECT id, name, email FROM users WHERE ...
     2  | 1, Alice, alice@example.com
     3  | 2, Bob, bob@example.com
     4  | 3, Carol, carol@example.com
     5  | 4, Dave, dave@example.com
... [1 823 lines truncated] ...
  1828  | 1827, Zara, zara@example.com
  1829  | 1828, Zoe, zoe@example.com
  1830  | 1829, ...
```

The agent can then call `read_file` with `offset` and `limit` to page through the full result on demand - reading only the portion that is relevant rather than loading everything into context at once.

The tools `ls`, `glob`, `grep`, `read_file`, `edit_file`, and `write_file` are excluded from eviction. They either have their own built-in truncation, return minimal confirmation text, or (for `read_file`) re-reading a truncated file would not help.

---

## Recap

- **SummarizationMiddleware** intercepts every LLM request via `wrap_model_call`; it runs before the model sees any messages
- When total tokens exceed 85% of `max_input_tokens`, old messages are partitioned at the **keep cutoff** and replaced with a summary
- The summary is generated by a separate LLM call using the same model as the agent
- `state["messages"]` is never mutated; summarization is tracked in a private `_summarization_event` field
- Passing `model.profile["max_input_tokens"]` overrides the threshold, making it practical to demo
- **FilesystemMiddleware** also evicts individual tool results that exceed ~20k tokens, offloading them to `/large_tool_results/` and replacing them with a preview + file reference
