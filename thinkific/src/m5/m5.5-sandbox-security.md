# Sandbox Security

<div markdown="1">

The slideshow below walks through why sandboxes exist, what they do and don't protect, and how to move data across the sandbox boundary safely.

<div class="sd-wrap" id="ss-sandbox"></div>

Starting from the Chinook Sales Assistant we built in 5.3, this lesson makes two changes: it moves chart generation into a sandbox, and it turns genre research into an async subagent. We'll build this in three parts:

1. **Move to a sandbox-backed filesystem.** The agent writes and runs its own charting code inside the sandbox, instead of calling a fixed-purpose tool. We'll also add a way to download whatever the agent produces there.
2. **Customize the chat UI.** The download button from part 1, plus a status indicator that shows which background research tasks are still running and which have finished.
3. **Make genre research async.** Genre research can take a long time, so we'll convert it into an async subagent — the main agent stays free to keep working while research runs in the background, and reports back once it's done.

Each part gets its own section below.

## Run it

```bash
cd python/m5/sales_assistant_sandbox
./start.sh
```

Same pattern as 5.3, plus the chat UI: `start.sh` launches the mock mail server, then the chat UI, then `langgraph dev` with a larger worker pool (`--n-jobs-per-worker 10` — see the note on that further down this page). On exit (Ctrl-C or otherwise), it stops all three, plus any sandboxes it created, so you're not paying for idle sandbox time between sessions.

5.3's hosted UI ([agentchat.vercel.app](https://agentchat.vercel.app)) doesn't have this lesson's sandbox-files panel or async-task badge — those are custom additions, so this lesson needs the self-hosted UI. That's the fork vendored at the repo root, in `agent-chat-ui/`, that `start.sh` launches automatically — it installs dependencies on first run, then serves the UI at `http://localhost:3000`, sourcing `LANGSMITH_API_KEY` from `python/.env` automatically. No separate command or `.env.local` key to manage.

## How genre research synchronizes with the main agent

Genre research runs as its own graph, not an in-process subagent. `AsyncSubAgentMiddleware` only talks to other graphs reachable through the LangGraph SDK, so `genre-researcher` becomes a second entry in `langgraph.json`, registered alongside the main agent in the same deployment.

**Launching is fire-and-forget.** When the main agent calls `start_async_task`, it creates a new thread on the `genre-researcher` graph, starts a run, and returns a task ID immediately. The main agent reports the ID and keeps working — it does not block, and it does not poll in a loop.

**The gap: nobody learns when a task finishes.** By default, the only way to learn a background task is done is to ask — call `check_async_task` or `list_async_tasks`. That's fine for short jobs, but it means the main agent (and the user) have to keep checking rather than being told.

**The fix: a completion notifier running on the subagent's side.** When a `genre-researcher` run finishes, a small middleware on that graph calls back into the main agent's own thread — the same LangGraph SDK mechanism the main agent used to launch it in the first place, just in the other direction. This queues a new run on the main agent's thread with a message like "genre-researcher finished: here's what it found." From the main agent's perspective, this looks exactly like a new message arriving — it wakes up and can act on it.

**Why this needs to know which thread to call back.** The `genre-researcher` graph is a single, shared object reused for every research task, from every conversation. It has no way to know, on its own, who launched a given run. So the main agent has to tell it: when it calls `start_async_task`, it passes along its own thread ID (and the name of its own graph). The genre-researcher run receives that as part of its configuration and uses it to address the callback correctly.

**Why the stock tool needs a small customization.** The `start_async_task` tool that ships with `AsyncSubAgentMiddleware` only takes a task description and an agent type — it has no field for "and also tell the subagent who I am." That's deliberate: how you'd pass that information is specific to each deployment. So this lesson adds one small custom version of that tool that does exactly one extra thing: attach the main agent's thread ID before launching.

**Why the genre-researcher graph gets built once per run, not once ever.** Normally a graph is built a single time and reused for every request. But the completion notifier needs to know *which* main-agent thread to call back for *this specific* run — and that's different every time. So instead of building the graph once with a fixed callback target, it gets rebuilt fresh for each run, reading the calling thread's ID out of that run's own configuration and wiring the notifier to point at the right place every time.

**Four research tasks finishing at once needs no special handling.** The weekly newsletter launches four genre-researcher tasks in parallel. Each one notifies independently when it's done, so the main agent can be woken up to four times. Each time it wakes up, it checks whether *all four* are finished — if not, it says so and waits again; once they are, it assembles the newsletter. Worst case, it says "waiting on the rest" a few times — nothing more coordinated is needed.

**What this doesn't do: push updates into an open browser tab.** The chat UI only shows runs it started itself; a run created by the completion notifier happens on the server and won't appear live in a tab that's just sitting open. The status indicator in the UI polls on its own short timer to show a running/done badge, but the assembled newsletter itself shows up the next time the user sends a message or reloads the page — not instantly. That's a deliberate trade-off for this lesson, not a limitation we tried and failed to solve.

**`langgraph dev` needs more worker slots for this to actually run concurrently.** Locally, `langgraph dev` starts with a queue of exactly one worker slot. Each active run — including each background subagent run — occupies a slot for its entire duration, not just while it's actively computing. Launch four genre-researcher tasks and they queue up and run one at a time behind that single slot, and the main agent's own thread gets stuck behind all four of them too, unable to respond to a new message until they've all cleared. A supervisor plus four concurrent subagents needs five slots at minimum. Start the server with `langgraph dev --n-jobs-per-worker 10` (already wired into this lesson's `start.sh`) to give it enough headroom. This is purely a local-dev artifact — a real deployment schedules runs with much higher concurrency — but without it, the async behavior this lesson builds will look broken when you try it yourself.

**Skills and `AGENTS.md` are seeded into the sandbox once, at creation — editing them on disk doesn't reach an existing thread.** `agent.py` copies `/skills/` and `/AGENTS.md` from local disk into a thread's sandbox the first time that thread's sandbox is created, and never again. Editing a `SKILL.md` file takes effect immediately for any **new** thread (a new sandbox gets seeded fresh from the current files on disk — no restart of `langgraph dev` needed, since skills aren't loaded into the server process at all). But an **existing** thread's sandbox already has its own copy from whenever it was created, and nothing re-syncs it — restarting `langgraph dev` doesn't help either, since the staleness lives in the sandbox's filesystem, not the server. To pick up a skill edit, start a new conversation.
