# The Sales Assistant

<div markdown="1">

Lesson 1 covered what the Chinook Sales Assistant is and which Deep Agents features each part uses. Lesson 2 covered serving an agent with `langgraph dev`. This lesson presents the finished assistant, examines how little new code it requires, and runs it locally.

The assistant works for Jane Peacock, a Sales Support Agent at Chinook (an online music distributor). "Her book of business" means the customers whose support rep is Jane (`Customer.SupportRepId = 3` in the database). The assistant helps her answer quote requests, keep customer records current, research the market, and report on her territory.

---

## How it fits together

Like any deployed agent, the assistant runs inside `langgraph dev` and is reached over HTTP by a chat UI. The distinguishing elements are what it draws on: the Chinook database, its sales playbooks, and mail.

## How little of it is new code

The project structure shows how much of the agent's behavior is defined by plain-text files rather than code:

```text
sales_assistant/
├── agent_no_sandbox.py # assembles the agent — the only "wiring" code
├── agent_sandbox.py    # sandbox variant — per-thread isolated execution
├── langgraph.json      # tells langgraph dev which graph to serve
├── langgraph_sandbox.json  # alternate config — used when ENABLE_SANDBOX=1
├── start.sh            # starts mail server + langgraph dev (selects config)
├── AGENTS.md           # operating manual - plain text, describes behavior
├── skills/             # one playbook per task - plain text, no code
│   ├── rfq-quote/
│   ├── weekly-newsletter/
│   └── territory-report/
├── subagents.py        # the four specialists
├── tools/              # tool code, each given to the agent that needs it
│   ├── sql.py          # → the chinook-analyst subagent
│   └── html.py         # → the main agent (builds HTML deliverables)
├── mcp/                # the mail MCP server — mail tools discovered here at startup
└── data/
    └── chinook.db      # the sales database (ships with the agent)
```

`AGENTS.md` and everything under `skills/` are natural language files, read directly by the model in any language it understands. The operating manual defines who the agent works for and how to behave; each skill is a playbook for one task. As covered in Lesson 1, most of the behavioral definition is natural language, not code.

## Run it

### Before you start

The assistant uses the shared `python/.env` from the course setup. Set `ENABLE_SANDBOX=1` to run `agent_sandbox.py` (isolated sandbox execution, required for charts) or leave it unset to run `agent_no_sandbox.py`. Add `TAVILY_API_KEY` to enable newsletter web research.

> **Note:** LangSmith Sandboxes require a LangSmith Plus account or higher.

### Running

The mock mail server must be running before the agent starts. `start.sh` launches the mail server, waits for it to accept connections, then starts `langgraph dev`.

```bash
cd python/m5/sales_assistant
./start.sh
```

This starts the server on `http://localhost:2024` and opens LangGraph Studio. Studio is useful for stepping through the graph, but the human-approval steps (saving a draft or adding a customer) are more convenient with `agent-chat-ui`, which renders those pauses as inline approve / edit / reject cards.

### Chat UI

LangChain hosts a ready-made chat interface with no install required:

1. Open [agentchat.vercel.app](https://agentchat.vercel.app).
2. Set Deployment URL to `http://localhost:2024` and Assistant / Graph ID to `agent`.
3. Chat as normal. When the assistant saves an email draft or adds a customer, an approve / edit / reject card appears right in the conversation.

You can also self-host from the [agent-chat-ui repo](https://github.com/langchain-ai/agent-chat-ui) with Node and `pnpm`, but the hosted version needs no setup.

### What it can do

The assistant supports three task sequences:

1. Process an incoming RFQ: read a quote request from the inbox, look up the customer and prices, compute the quote exactly with the interpreter, have a reviewer subagent check it, draft the reply for approval, and log it. Try: `Check the inbox for any quote requests and process them.`
2. Write the weekly newsletter: fan out parallel research subagents (one per genre), then assemble an HTML newsletter on disk. *(requires `TAVILY_API_KEY`)* Try: `Write this week's "This Week in Music" newsletter.` Output is saved to `python/m5/sales_assistant/outputs/newsletter-<date>.html`.
3. Report on the territory: pull metrics from the database and, with the sandbox enabled, render a chart. *(add `ENABLE_SANDBOX=1` to include a chart)* Try: `How's my book of business looking? Give me a territory report.` Report is saved to `python/m5/sales_assistant/outputs/territory_report-<date>.md`; chart (if sandbox enabled) to `outputs/territory_chart.png`.

### Resetting the lab

The mock inbox/drafts and the SQLite database can change as you run the assistant. To replay the RFQ flow, rerun the inbox seed helper from `python/m5/sales_assistant`:

```bash
uv run python mcp/send_to_inbox.py --reset
```

If you approve an `add_customer` action, the local `data/chinook.db` file is modified. Restore it from git before rerunning the new-customer path if you want a clean starting point.

### Under the hood

`start.sh` runs one of two agent files depending on `ENABLE_SANDBOX`: `agent_no_sandbox.py` (local filesystem, QuickJS interpreter) or `agent_sandbox.py` (per-thread LangSmith sandbox, required for charts). The no-sandbox version is below; the sandbox variant follows.

Both configs point `langgraph dev` at a factory function in the chosen agent file (`python/m5/sales_assistant/langgraph.json`):

```json
{
  "dependencies": ["../.."],
  "graphs": {
    "agent": "./agent_no_sandbox.py:make_graph"
  },
  "env": "../../.env"
}
```

Unlike Lesson 2's lab, this points at an async factory function. `langgraph dev` calls `make_graph()` on startup and serves the graph it returns, allowing `create_deep_agent` to run after the mail tools have been discovered from the MCP server.

```python {16,32,44,46,49}
# python/m5/sales_assistant/agent_no_sandbox.py
"""Chinook Sales Assistant — no-sandbox variant.

Uses a local FilesystemBackend and QuickJS code interpreter.

Start with:
    ./start.sh
"""

from __future__ import annotations

import logging
import os
from pathlib import Path

from deepagents import create_deep_agent
from deepagents.backends import FilesystemBackend
from langchain_mcp_adapters.client import MultiServerMCPClient
from langchain_quickjs import CodeInterpreterMiddleware
from subagents import build_subagents
from tools.html import markdown_to_html

from models import strong_model

logger = logging.getLogger(__name__)

HERE = Path(__file__).resolve().parent

SYSTEM_PROMPT = (
    "You are a sales assistant for Jane Peacock, a Sales Support Agent at "
    "Chinook, an online music distributor. Follow your operating manual (loaded "
    "from your memory) and use the matching playbook from /skills/ for each task."
)

MAIL_SERVER = {"transport": "streamable-http", "url": "http://127.0.0.1:5002/mcp"}

_enable_search = bool(os.environ.get("TAVILY_API_KEY"))
if not _enable_search:
    logger.info("TAVILY_API_KEY not set — newsletter research subagent disabled.")

_backend = FilesystemBackend(root_dir=str(HERE), virtual_mode=True)


async def make_graph():
    client = MultiServerMCPClient({"mock-mail": MAIL_SERVER})
    mail_tools = await client.get_tools()
    return create_deep_agent(
        model=strong_model,
        tools=[markdown_to_html] + mail_tools,
        system_prompt=SYSTEM_PROMPT,
        subagents=build_subagents(_backend, enable_search=_enable_search, mail_tools=mail_tools),
        skills=["/skills"],
        memory=["/AGENTS.md"],
        backend=_backend,
        middleware=[CodeInterpreterMiddleware()],
        name="chinook-sales-assistant",
    )
```

### Sandbox: The code that wires it together

The sandbox variant differs from the no-sandbox version in five ways:

- `@contextlib.asynccontextmanager` makes the factory a context manager: the server sets up resources before yielding the graph and cleans them up afterward.
- The factory accepts a `ServerRuntime` parameter and checks `runtime.execution_runtime` to decide whether to spin up a sandbox at all (non-None only for real runs).
- A per-thread `LangSmithSandbox` wrapped in a `CompositeBackend` routes code execution to the sandbox while keeping skills and memory on local disk.
- A `retrieve_output` tool copies sandbox files to local `outputs/`.
- `yield` replaces `return`: local Python objects are released when the context manager exits, but the sandbox persists on LangSmith, cleaned up by its idle TTL rather than by the context manager.
- `_lookup_or_create` (shown in abbreviated form below) also restarts a sandbox that has gone idle rather than only reusing or creating one.

`langgraph_sandbox.json`:

```json
{
  "dependencies": ["../.."],
  "graphs": {
    "agent": "./agent_sandbox.py:make_graph"
  },
  "env": "../../.env"
}
```

<Image src="images/m5.3-make-graph-sandbox-anatomy.svg" width="840px" alt="Diagram calling out four parts of agent_sandbox.py: @contextlib.asynccontextmanager plus async def make_graph(config, runtime) (context manager — server injects RunnableConfig and ServerRuntime by type annotation), if ert := runtime.execution_runtime (execution check — non-None only for real runs; introspection calls skip the sandbox entirely), @tool def retrieve_output (file bridge — write to /retrieve/ in the sandbox and call retrieve_output to copy to outputs/), and yield create_deep_agent(...) (yield not return — context manager releases sandbox on cleanup)." />

Here is the core of `agent_sandbox.py` (the shipped file's `_lookup_or_create` also retries a sandbox that has gone idle before falling back to creating a new one):

<div id="agentpy-sandbox-scroll" markdown="1">

```python {97,98,101,105,106,114,115,143}
# python/m5/sales_assistant/agent_sandbox.py
"""Chinook Sales Assistant — sandbox variant.

Uses a LangSmith sandbox as the execution backend so the agent can install
and run packages (e.g. matplotlib for charts) natively via code execution.
All filesystem operations (skills, AGENTS.md, working files) stay on local
disk via a CompositeBackend that routes "/" to FilesystemBackend.

Each LangGraph thread gets its own sandbox (thread-scoped), looked up by
name so follow-up messages in the same thread reuse the same environment.
Sandboxes are cleaned up automatically by a 10-minute idle TTL.

Start with:
    ENABLE_SANDBOX=1 ./start.sh
"""

from __future__ import annotations

import asyncio
import contextlib
import logging
import os
from pathlib import Path

from deepagents import create_deep_agent
from deepagents.backends import FilesystemBackend
from deepagents.backends.composite import CompositeBackend
from deepagents.backends.langsmith import LangSmithSandbox
from langchain_core.runnables import RunnableConfig
from langchain_core.tools import tool
from langchain_mcp_adapters.client import MultiServerMCPClient
from langgraph_sdk.runtime import ServerRuntime
from subagents import build_subagents
from tools.html import markdown_to_html

from models import strong_model

logger = logging.getLogger(__name__)

HERE = Path(__file__).resolve().parent

SYSTEM_PROMPT = (
    "You are a sales assistant for Jane Peacock, a Sales Support Agent at "
    "Chinook, an online music distributor. Follow your operating manual (loaded "
    "from your memory) and use the matching playbook from /skills/ for each task.\n\n"
    "Code runs in an isolated sandbox container. Files you create there (charts, "
    "reports, exports) are not visible on the local filesystem. To make a file "
    "available to the user, write it to /retrieve/<filename> in the sandbox, then "
    "call retrieve_output to copy it to the local outputs/ directory."
)

MAIL_SERVER = {"transport": "streamable-http", "url": "http://127.0.0.1:5002/mcp"}

_enable_search = bool(os.environ.get("TAVILY_API_KEY"))
if not _enable_search:
    logger.info("TAVILY_API_KEY not set — newsletter research subagent disabled.")

def _lookup_or_create(name: str):
    from langsmith.sandbox import SandboxClient
    client = SandboxClient()
    existing = [s for s in client.list_sandboxes() if s.name == name]
    if existing:
        logger.info("Reusing sandbox %s", name)
        return existing[0]
    try:
        sb = client.create_sandbox(name=name, idle_ttl_seconds=600)
        logger.info("Created sandbox %s", name)
        return sb
    except Exception:
        # Another thread created it between our list and our create — look it up
        existing = [s for s in client.list_sandboxes() if s.name == name]
        if existing:
            logger.info("Reusing sandbox %s (race recovery)", name)
            return existing[0]
        raise


# Thread-scoped sandbox pattern:
# https://docs.langchain.com/langsmith/graph-rebuild#context-manager-factory
#
# The factory accepts ServerRuntime so the server can signal whether it is
# processing an actual run (execution_runtime is non-None) or handling
# introspection calls (get_schema, get_graph, assistants.read, …). When
# execution_runtime is None we skip the expensive sandbox setup and fall back
# to local filesystem — same graph topology, no sandbox, no MCP round-trip.
# Real runs get their own thread-scoped sandbox looked up by thread_id.
@contextlib.asynccontextmanager
async def make_graph(config: RunnableConfig, runtime: ServerRuntime):
    ls_backend = None

    if runtime.execution_runtime:
        # Real run — look up or create a thread-scoped sandbox
        thread_id = config.get("configurable", {}).get("thread_id")
        sandbox = await asyncio.to_thread(_lookup_or_create, f"thread-{thread_id}")
        ls_backend = LangSmithSandbox(sandbox)
        backend = CompositeBackend(
            default=ls_backend,
            routes={"/": FilesystemBackend(root_dir=str(HERE), virtual_mode=True)},
        )
    else:
        # Introspection — same graph shape, no sandbox, no MCP round-trip
        backend = FilesystemBackend(root_dir=str(HERE), virtual_mode=True)

    @tool
    def retrieve_output(sandbox_path: str) -> str:
        """Copy a file from the sandbox to the local outputs/ directory.

        The sandbox runs in an isolated container — files saved there are not
        visible on the local filesystem until explicitly retrieved. To make a
        file available, write it to /retrieve/<filename> inside the sandbox,
        then call this tool to copy it to outputs/<filename> on local disk
        where the user can open it. Files only; directories are not supported.

        Args:
            sandbox_path: File path inside the sandbox, e.g. /retrieve/chart.png
        """
        if ls_backend is None:
            return "retrieve_output is not available in this context."
        responses = ls_backend.download_files([sandbox_path])
        resp = responses[0]
        if resp.content is None:
            return f"Error retrieving {sandbox_path}: {resp.error}"
        # Use only the filename to prevent path traversal on local disk
        filename = Path(sandbox_path).name
        out_path = HERE / "outputs" / filename
        out_path.parent.mkdir(exist_ok=True)
        out_path.write_bytes(resp.content)
        return f"Saved to outputs/{filename}"

    mcp_client = MultiServerMCPClient({"mock-mail": MAIL_SERVER})
    mail_tools = await mcp_client.get_tools()

    yield create_deep_agent(
        model=strong_model,
        tools=[markdown_to_html, retrieve_output] + mail_tools,
        system_prompt=SYSTEM_PROMPT,
        subagents=build_subagents(backend, enable_search=_enable_search, mail_tools=mail_tools),
        skills=["/skills"],
        memory=["/AGENTS.md"],
        backend=backend,
        name="chinook-sales-assistant",
    )
```

> **Note:** `retrieve_output` is a closure: it captures `ls_backend` at the time the graph is built. When `runtime.execution_runtime` is `None` (introspection calls), `ls_backend` stays `None` and the tool returns early without attempting a download. See [Context manager factory](https://docs.langchain.com/langsmith/graph-rebuild#context-manager-factory) in the LangSmith docs for the full `ServerRuntime` reference.

### Using the Sandbox

Each thread gets its own sandbox, named `thread-{thread_id}`. Follow-up messages in the same thread reuse it via `_lookup_or_create`; a fresh sandbox is only created when none exists. The sandbox has an idle TTL of 10 minutes. After 10 minutes of inactivity, LangSmith stops it; the next message restarts the same sandbox rather than creating a new one.

The agent invokes the code interpreter like any other tool, passing code as a string argument:

```
execute_code(
    code="""
import matplotlib.pyplot as plt
genres = ["Rock", "Jazz", "Classical", "Pop"]
revenues = [4200, 1800, 900, 3100]
plt.bar(genres, revenues)
plt.savefig('/retrieve/territory_chart.png')
"""
)
```

The sandbox can install packages and run arbitrary code, but it writes to its own filesystem. Files do not appear on the local machine automatically.

Because the sandbox is the default backend, `AGENTS.md` and skills would land there too without routing, and be lost on TTL cleanup. `CompositeBackend` fixes this by redirecting `"/"` to `FilesystemBackend`, keeping memory and skills on local disk.

Sandbox code writes to its local filesystem; output files are written to `/retrieve/<filename>`. To retrieve them, the agent calls the `retrieve_output` tool, which downloads the file from the sandbox and saves it to `outputs/`. This approach has two benefits: the file never passes through the model, avoiding wasted context on binary data, and potentially unsafe sandbox output is kept out of the LLM's context window.
