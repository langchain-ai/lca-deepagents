In Lesson 1 you saw *what* the Chinook Sales Assistant is and which Deep Agents feature each part uses. In Lesson 2 you learned how to *serve* an agent with `langgraph dev`. Now we put it together: meet the finished assistant, see how little *new* code you write to assemble it, and run it locally.

The assistant works for **Jane Peacock**, a Sales Support Agent at Chinook (an online music distributor). "Her book of business" means the customers whose support rep is Jane - `Customer.SupportRepId = 3` in the database. The assistant helps her answer quote requests, keep customer records current, research the market, and report on her territory.

> **Note:** This lesson sets the assistant up and runs it. The three task sequences it can perform are demonstrated in the video walkthrough above.

## How it fits together

Like any deployed agent, the assistant runs **inside `langgraph dev`** and is reached over HTTP by a chat UI. What's new here is everything the agent draws on: the **Chinook database**, its **sales playbooks**, and **mail**.

## How little of it is *new* code

Here is the project. There *is* real code here - the tools, the subagents, a mock mail server. But notice how much of the agent's **behavior** is set by plain-text files rather than by new code you write:

```text
sales_assistant/
├── agent.py            # assembles the agent - the only "wiring" code
├── langgraph.json      # tells langgraph dev which graph to serve
├── AGENTS.md           # operating manual - plain text, describes behavior
├── skills/             # one playbook per task - plain text, no code
│   ├── rfq-quote/
│   ├── weekly-newsletter/
│   └── territory-report/
├── subagents.py        # the four specialists
├── tools/              # tool code, each given to the agent that needs it
│   ├── sql.py          # → the chinook-analyst subagent
│   └── html.py         # → the main agent (builds HTML deliverables)
├── mail_mcp.py         # discovers the mail tools
├── mcp/                # the local mock mail server + a seed inbox
└── data/
    └── chinook.db      # the sales database (ships with the agent)
```

`AGENTS.md` and everything under `skills/` are just **natural language** - read directly by the model, in any language it understands. The operating manual sets out who the agent works for and how to behave; each skill is a playbook for one task. For example, `AGENTS.md` states the approval rules in prose:

```text
## Approvals (human-in-the-loop)

Two actions wait for Jane to approve, edit, or reject before they take effect:

- Saving an email draft (inbox-manager): drafts are never sent, only saved.
- Adding a new customer (chinook-analyst): no row is written without her ok.
```

That is Lesson 1's point: much of the work is describing behavior, not writing new code.

### The code that wires it together

You still write code - your tools (`tools/`) and the specialists (`subagents.py`). But the *assembly* is tiny: `agent.py` pulls those pieces together and hands the result to the server in a single `create_deep_agent(...)` call.

Here is the important part of the file, centered on `make_graph`, the factory that assembles the agent. The source file also includes imports, logging setup, and comments around mail discovery:

<div id="agentpy-scroll" markdown="1">

```python {59,74}
# python/m5/sales_assistant/agent.py
"""The Chinook Sales Assistant - served as a graph for `langgraph dev`.

This module is just the agent. The mail MCP plumbing lives in `mail_mcp.py`;
all this file does is assemble the agent around whatever tools that discovery
returns.

`langgraph.json` points at the async factory `make_graph`, which langgraph awaits
once at startup. We use a factory (not a plain module-level `agent = ...`) for
one reason: the mail tools come from an MCP server, and asking an MCP server for
its tool list is an async call - so we discover the tools up front, then build
the agent around whatever we found.
"""

from __future__ import annotations

import logging
import os
from pathlib import Path

from deepagents import create_deep_agent
from deepagents.backends import FilesystemBackend
from mail_mcp import load_mail_tools
from langchain_quickjs import CodeInterpreterMiddleware
from subagents import build_subagents
from tools.html import markdown_to_html

from models import strong_model

logger = logging.getLogger("chinook-sales-assistant")

HERE = Path(__file__).resolve().parent


SYSTEM_PROMPT = (
    "You are a sales assistant for Jane Peacock, a Sales Support Agent at "
    "Chinook, an online music distributor. Follow your operating manual (loaded "
    "from your memory) and use the matching playbook from /skills/ for each task."
)


async def make_graph():
    """Async graph factory - built once by `langgraph dev` at startup."""
    # Real local disk under this module only (virtual_mode locks the agent to
    # this directory: skills/, AGENTS.md, outputs/). Suitable for local dev;
    # not for a multi-tenant web server.
    backend = FilesystemBackend(root_dir=str(HERE), virtual_mode=True)

    enable_search = bool(os.environ.get("TAVILY_API_KEY"))
    if not enable_search:
        logger.info("TAVILY_API_KEY not set - newsletter research subagent disabled.")

    mail_tools = await load_mail_tools()

    # The main agent holds NO gated tools. mail_create_draft lives on
    # the inbox-manager subagent and SQL writes (add_customer) on chinook-analyst,
    # each with its own approval gate - so the always-present general-purpose
    # subagent, which inherits the main agent's tools, can never reach a write
    # ungated. The main agent keeps only safe tools plus the interpreter.
    tools = [markdown_to_html]

    # Optional sandbox-backed chart tool (the course's optional feature).
    if os.environ.get("ENABLE_SANDBOX"):
        from tools.chart import render_chart

        tools.append(render_chart)

    return create_deep_agent(
        model=strong_model,
        tools=tools,
        system_prompt=SYSTEM_PROMPT,
        subagents=build_subagents(
            backend, enable_search=enable_search, mail_tools=mail_tools
        ),
        skills=["/skills"],
        memory=["/AGENTS.md"],
        backend=backend,
        middleware=[CodeInterpreterMiddleware()],
        name="chinook-sales-assistant",
    )
```

> **Note:**

- **You hand the server a *graph*, not a `main()`.** Earlier modules ran an agent with `asyncio.run(main())`. Here you never run it yourself: `langgraph.json` points at `make_graph`, and `langgraph dev` calls it. You *expose* the agent; the server runs it.
- **You discover the mail tools *before* building the agent.** A model can only call a tool it has been told about, and an MCP server hands over its tool list through an async call. So you `await load_mail_tools()` first, then build the agent around whatever came back.

And `subagents.py`, which defines the four specialists:

```python {17,29,36,40}
# python/m5/sales_assistant/subagents.py

def build_subagents(
    backend: BackendProtocol,
    *,
    enable_search: bool,
    mail_tools: list | None = None,
) -> list[dict]:

    chinook_analyst = {
        "name": "chinook-analyst",
        "description": "Query the Chinook database ...",
        "system_prompt": ANALYST_PROMPT,
        "tools": [query_chinook, introspect_schema, add_customer],
        "model": model,
        "middleware": [MemoryMiddleware(backend=backend, sources=["/agents/chinook-analyst/AGENTS.md"])],
        "interrupt_on": {"add_customer": _APPROVE_EDIT_REJECT},
    }

    quote_reviewer = {
        "name": "quote-reviewer",
        "description": "Review a drafted quote ...",
        "system_prompt": REVIEWER_PROMPT,
        "model": strong_model,
    }

    subagents = [chinook_analyst, quote_reviewer]

    if mail_tools:
        inbox_manager = {
            "name": "inbox-manager",
            "description": "Read Jane's inbox ...",
            "system_prompt": INBOX_PROMPT,
            "tools": list(mail_tools),
            "model": model,
            "interrupt_on": {"mail_create_draft": _APPROVE_EDIT_REJECT},
        }
        subagents.append(inbox_manager)

    if enable_search:
        genre_researcher = {
            "name": "genre-researcher",
            "description": "Research one music genre ...",
            "system_prompt": GENRE_PROMPT,
            "tools": [internet_search],
            "model": model,
            "permissions": [...],
        }
        subagents.append(genre_researcher)

    return subagents
```

The highlighted lines show the two gated writes (`interrupt_on`) and the two subagents that are conditional — inbox-manager only when mail tools were discovered, genre-researcher only when web search is configured.

And the `langgraph.json` that points at that factory (`python/m5/sales_assistant/langgraph.json`):

```json
{
  "dependencies": ["../.."],
  "graphs": {
    "agent": "./agent.py:make_graph"
  },
  "env": "../../.env"
}
```

Compare this to Lesson 2's lab, where the graph was a plain variable (`"./agent.py:graph"`). Here it points at a **function** (`make_graph`) that the server awaits - that's how we get to run async tool discovery at startup.

## Run it

Serve the assistant the same way you served the Lesson 2 agent - with one extra flag:

```bash
cd python/m5/sales_assistant
uv run langgraph dev --allow-blocking
```

Why `--allow-blocking`? Discovering the mock mail server's tools launches a subprocess at startup - synchronous I/O that `langgraph dev`'s blocking-call detector would otherwise reject.

This starts the server on `http://localhost:2024` and opens **LangGraph Studio**. Studio is useful for stepping through the graph, but the human-approval steps — saving a draft or adding a customer — require **agent-chat-ui**, which renders those pauses as inline approve / edit / reject cards.

### Chat UI (required for human approvals)

LangChain hosts a ready-made chat interface - no install required:

1. Open **[agentchat.vercel.app](https://agentchat.vercel.app)**.
2. Set **Deployment URL** to `http://localhost:2024` and **Assistant / Graph ID** to `agent`.
3. Chat as normal. When the assistant saves an email draft or adds a customer, an **approve / edit / reject** card appears right in the conversation.

> **Note:** Studio's interrupt prompt does not speak the same protocol as the approval middleware. Use agent-chat-ui for this lab.

You can also self-host from the [agent-chat-ui repo](https://github.com/langchain-ai/agent-chat-ui) with Node and `pnpm`, but the hosted version needs no setup.

### Config knobs

The assistant reads its options from the shared `python/.env`. With the standard course setup, the RFQ flow and numeric territory report run with the local mock mail server and no sandbox. Optional knobs:

- `ENABLE_SANDBOX`: add it (e.g. `ENABLE_SANDBOX=1`) to turn on the territory chart. The chart-rendering tool uses a LangSmith Sandbox; the main agent still uses its local filesystem backend. Leave it out to run without a chart.
- `TAVILY_API_KEY`: enables the newsletter's web research. Without it, the newsletter researcher is disabled; ask a different task or expect the assistant to report that web research is unavailable.

## What it can do

With the assistant running, you can ask it to handle a sales rep's day-to-day work. The video walks through three **task sequences**, each showing off different Deep Agents features:

1. **Process an incoming RFQ**: read a quote request from the inbox, look up the customer and prices, compute the quote exactly with the interpreter, have a reviewer subagent check it, draft the reply for approval, and log it.
2. **Write the weekly newsletter**: fan out parallel research subagents (one per genre), then assemble an HTML newsletter on disk.
3. **Report on the territory**: pull metrics from the database and, with the sandbox enabled, render a chart.

Try them yourself once the server is up. To start each task:

- **RFQ**: `Check the inbox for any quote requests and process them.`
- **Newsletter**: `Write this week's "This Week in Music" newsletter.` *(requires `TAVILY_API_KEY`)*
- **Territory report**: `How's my book of business looking? Give me a territory report.` *(add `ENABLE_SANDBOX=1` to include a chart)*

### Resetting the lab

The mock inbox/drafts and the SQLite database can change as you run the assistant. To replay the RFQ flow, rerun the inbox seed helper from `python/m5/sales_assistant`:

```bash
uv run python mcp/send_to_inbox.py --reset
```

If you approve an `add_customer` action, the local `data/chinook.db` file is modified. Restore it from git before rerunning the new-customer path if you want a clean starting point.

## Recap

In this lesson you met and ran the capstone:

- The **Chinook Sales Assistant** runs inside `langgraph dev` and draws on the **Chinook database**, its **sales playbooks**, and **mail**.
- Much of the agent's behavior lives in **plain text**: `AGENTS.md` and the `skills/` playbooks, written in natural language. The code you write is small and focused (your tools and subagents), wired together by a single `create_deep_agent(...)` call.
- The agent is served as a **graph** (a `make_graph` factory), not run with `main()`, and it **discovers its MCP tools up front**: tolerating a down mail server.
- Run it with `uv run langgraph dev --allow-blocking` and chat via **agent-chat-ui** (agentchat.vercel.app); the **two approval steps** (saving a draft, adding a customer) pause and render as inline approve / edit / reject cards. Studio is useful for graph inspection but does not support the approval protocol.
