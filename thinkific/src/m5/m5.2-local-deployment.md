### Studio

`langgraph dev` includes a chat and debugging interface called "Studio". As shown in the diagram, Studio uses the API endpoints provided by `langgraph dev` to allow you to converse with your agent via chat as well as examine and control the underlying agentic loop via a graphical debugger.

## The agent's API

Every agent served by `langgraph dev` exposes the same **Agent Server API** ([full reference](https://docs.langchain.com/langsmith/server-api-ref)). Because it's a running web service, the dev server documents itself: once it's up, open **`http://localhost:2024/docs`** in your browser to see the live, interactive API for your agent.

The endpoints group into a handful of categories. Here are the ones you'll use most, with the advanced ones grouped below.

<details>
<summary><strong>Assistants</strong> (a configured instance of a graph)</summary>

<br>

The deployable unit: a graph plus its configuration.

- `POST /assistants`
- `POST /assistants/search`
- `GET /assistants/{id}/graph`
- `GET /assistants/{id}/schemas`

</details>

<details>
<summary><strong>Threads</strong> (the accumulated state of a conversation)</summary>

<br>

A thread holds the accumulated outputs of a group of runs, forming the persistent conversation.

- `POST /threads`
- `GET /threads/{id}/state`
- `GET /threads/{id}/history`
- `POST /threads/search`

</details>

<details>
<summary><strong>Thread Runs</strong> (one invocation of the agent on a thread)</summary>

<br>

A run invokes the graph on a thread and updates its state.

- `POST /threads/{id}/runs`
- `POST /threads/{id}/runs/stream`
- `POST /threads/{id}/runs/wait`
- `POST /threads/{id}/runs/{run_id}/cancel`

</details>

<details>
<summary><strong>Store</strong> (long-term key-value memory)</summary>

<br>

A persistent key-value store (long-term memory) available from any thread.

- `PUT /store/items`
- `GET /store/items`
- `POST /store/items/search`
- `GET /store/namespaces`

</details>

<details>
<summary>Advanced (Stateless Runs, Crons, Streaming, A2A, MCP, System)</summary>

<br>

- **Stateless Runs**: a run with no state or memory persistence: `POST /runs`, `POST /runs/stream`, `POST /runs/wait`, `POST /runs/batch`
- **Crons**: a periodic run on a schedule: `POST /runs/crons`, `POST /threads/{id}/runs/crons`
- **Streaming**: thread-centric streaming over SSE+HTTP (WebSocket also supported): `POST /threads/{id}/stream`, `/threads/{id}/stream/events`
- **A2A**: expose an assistant as an A2A-compliant agent: `POST /a2a/{assistant_id}`
- **MCP**: expose the agent as an MCP server: `/mcp/`
- **System**: health, metrics, server info: `GET /ok`, `GET /info`, `GET /metrics`, `GET /docs`

</details>

`langgraph dev` is intended for local development. Deploying an agent to production as a managed Cloud service, a standalone server, or a fully self-hosted platform introduces additional infrastructure and operational considerations beyond the scope of this lesson. The following resources cover those options in more detail:

## Recap

In this lesson you saw what it means to *deploy* an agent locally:

- A deployed agent runs as a **service reached over HTTP**, not a script you `invoke()` in Python.
- **`langgraph dev`** hosts your agent, exposes it as an **API**, and uses a small **`langgraph.json`** config file to declare which graph to serve.
- The **Agent Server API** is the same for every agent: **Assistants**, **Threads**, **Thread Runs**, and **Store**, plus advanced endpoints. The running server documents itself at **`localhost:2024/docs`**.
- **`langgraph dev` ships LangGraph Studio** as a client for that API, a combined **chat + debug** interface. Any other chat UI reaches the agent the same way, over HTTP.

## Lab: Deploy a Basic Agent

<details style="border:2.5px solid #000;border-radius:6px;background:#fff;margin:1rem 0;"><summary style="padding:10px 16px;cursor:pointer;font-weight:500;font-family:'IBM Plex Mono',monospace;font-size:14px;">Video Walkthrough</summary><div style="padding:12px 16px 16px;"><div class="video-container" style="max-width:750px;"><div class="video-wrapper"><iframe src="https://share.descript.com/embed/kVx79ZeFLVY" frameborder="0" allow="autoplay; fullscreen; encrypted-media; picture-in-picture" allowfullscreen></iframe></div></div></div></details>

You'll build a simple agent.

The agent itself is a single `create_deep_agent` call. It pulls `model` from the course's shared `models.py`, so the agent uses whatever provider you configured during setup, and is exposed as a module-level `graph` so the server can find it:

```python
# python/m5/hello/agent.py
from deepagents import create_deep_agent
from models import model

# langgraph.json points at this variable: "./agent.py:graph"
graph = create_deep_agent(model=model)
```

Next to it, a **`langgraph.json`** tells `langgraph dev` which graph to serve, where its dependencies are, and which env file to load (`python/m5/hello/langgraph.json`):

```json
{
  "dependencies": ["../.."],
  "graphs": {
    "agent": "./agent.py:graph"
  },
  "env": "../../.env"
}
```

That's the entire deployment: one graph, declared in one config file. Now serve it:

Before running it, make sure your model API key from setup is available in the Python environment.

```bash
cd python/m5/hello
uv run langgraph dev
```

The server starts on `http://localhost:2024` and **opens LangGraph Studio in your browser** automatically. In Studio you can:

- **Chat with the agent** in the conversation panel: type a message and watch it reply.
- **Walk through the state machine**: the graph view shows each node; step through a run to see state change as the agent works.
