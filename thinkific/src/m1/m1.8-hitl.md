In the tools lesson, you saw that tools can do more than retrieve information: they can also take actions. In the previous lesson, you saw how messages, threads, and checkpointers let an agent save and resume state. Human-in-the-Loop (HITL) combines those ideas: pause before a sensitive action and let a person approve, edit, reject, or respond to it.

The main reason is safety. If an agent is about to send an email, update a database, or delete a file, you may want a human checkpoint before the side effect happens. HITL also lets the agent suspend execution while it waits instead of staying active indefinitely.

---

## Decision types

<details style="border:2.5px solid #000;border-radius:6px;background:#fff;margin:1rem 0;"><summary style="padding:10px 16px;cursor:pointer;font-weight:500;font-family:'IBM Plex Mono',monospace;font-size:14px;">Video Walkthrough: Decision Types</summary><div style="padding:12px 16px 16px;"><div class="video-container" style="max-width:750px;"><div class="video-wrapper"><iframe src="https://share.descript.com/embed/KZb6UyoCWiX" frameborder="0" allow="autoplay; fullscreen; encrypted-media; picture-in-picture" allowfullscreen></iframe></div></div></div></details>

The `allowed_decisions` list controls what actions a human can take when reviewing a tool call:

<table>
<thead>
<tr><th>Decision Type</th><th>Description</th><th>Example Use Case</th></tr>
</thead>
<tbody>
<tr><td><code>approve</code></td><td>Execute the tool with the original arguments as proposed by the agent.</td><td>Send an email draft exactly as written</td></tr>
<tr><td><code>edit</code></td><td>Modify the tool arguments before execution.</td><td>Change the recipient before sending an email</td></tr>
<tr><td><code>reject</code></td><td>Skip executing this tool call entirely and return rejection feedback to the agent.</td><td>Deny file deletion and explain why</td></tr>
<tr><td><code>respond</code></td><td>Return the human's message directly as a synthetic tool result, skipping execution, for "ask user" style tools.</td><td>The agent calls <code>ask_user("What's your budget?")</code>; the human types "$500", and that becomes the tool's result.</td></tr>
</tbody>
</table>

`reject` blocks an action the tool would otherwise perform, like deleting a file.

`respond` is different: it's for tools where the human's answer *is* the result, like `ask_user`. Never use `respond` to block a side-effecting tool such as `send_email` or `delete_file`; the model will read the human's message as a successful result, not a denial, and may believe the action already happened.

<Tip>

When **editing** tool arguments, make changes conservatively. Significant modifications to the original arguments may cause the model to re-evaluate its approach and potentially execute the tool multiple times or take unexpected actions.

</Tip>

Open `python/m1/m1.8_hitl.py`. It defines a mock `send_email` tool. The tool does not contact a real email service; it returns a string so the lab is safe to run.

### 1. Define an action tool

```python
from langchain_core.tools import tool

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """Send an email to a recipient."""
    return f"Email sent to {to} with subject {subject!r}. Body: {body}"
```

### 2. Configure HITL on that tool

```python
from deepagents import create_deep_agent
from langgraph.checkpoint.memory import MemorySaver

agent = create_deep_agent(
    model=model,
    tools=[send_email],
    system_prompt=SYSTEM_PROMPT,
    interrupt_on={"send_email": True},
    checkpointer=MemorySaver(),
)
```

`interrupt_on={"send_email": True}` means: if the agent tries to call `send_email`, pause before the tool executes and surface the pending action for review.

The checkpointer is required because the agent needs somewhere to save the paused state. Without it, there is nothing to resume.

### 3. Invoke with a stable thread ID

```python
config = {"configurable": {"thread_id": "m1-7-hitl-demo"}}

result = agent.invoke(
    {
        "messages": [
            {
                "role": "user",
                "content": "Send an email to jane@example.com saying I will be 10 minutes late.",
            }
        ]
    },
    config=config,
    version="v2",
)
```

The `thread_id` is how the checkpointer finds this paused run again. When you resume, pass the same `config`.

### 4. Review and resume

```python
from langgraph.types import Command

while result.interrupts:
    pending = result.interrupts[0].value
    decisions = []
    for req in pending["action_requests"]:
        print(f"\nApproval required for {req['name']}:")
        print(req["args"])

        choice = input("\nApprove, edit, or reject? (approve/edit/reject): ").strip().lower()
        if choice in ("approve", "yes", "y"):
            decisions.append({"type": "approve"})
        elif choice in ("edit", "e"):
            edited_args = dict(req["args"])
            edited_args["body"] = input("New email body: ")
            decisions.append(
                {
                    "type": "edit",
                    "edited_action": {"name": req["name"], "args": edited_args},
                }
            )
        else:
            decisions.append(
                {"type": "reject", "message": "User rejected this email draft."}
            )

    result = agent.invoke(Command(resume={"decisions": decisions}), config=config, version="v2")
```

If you approve, `send_email` runs as proposed. If you edit, the tool runs with your revised arguments. If you reject, the tool never runs; the agent receives a cancellation result and should respond accordingly.

### Run it

From the repo root:

```bash
cd python
uv run python m1/m1.8_hitl.py
```

Try all three paths:

- run once and type `approve`;
- run again and type `edit`, then provide a new email body;
- run again and type `reject`.

You should see the pending tool arguments before each decision. That is the safety checkpoint: the side effect is visible before it happens.

The first lab reviewed a single tool with `approve`, `edit`, and `reject`. This lab adds a second tool, `ask_user`, whose only allowed decision is `respond`.

All four decision types live on the same `allowed_decisions` list; a tool is not "a respond tool" or "an approve/edit/reject tool" by nature. `ask_user` is restricted to `respond` here because that is the only decision that makes sense for a tool whose entire job is answering a question, not because `respond` belongs to some separate system.

**Scenario:** "Email professor.weng@university.edu that I'll be late, but first check whether I should just apologize or ask for a short extension." The agent cannot draft a good email until it knows which one you want, so it calls `ask_user` before it ever calls `send_email`.

<Tip>

In the trace, look at the `ask_user` tool call: it never actually runs. `respond` simulates the tool call for it, and the tool's output in the trace is not the placeholder string from the function body, it's the human's answer, inserted as if the tool had returned it.

</Tip>

<Tip>

**Why not just put this in the system prompt?** You could instruct the model to "ask the user to pick an option before continuing." The difference is enforcement. A system prompt only shapes what the model *tends* to do; it can still skip the question, phrase it as free text, or drift off-script.

`allowed_decisions=["respond"]` is enforced by the framework itself: the tool call always pauses, and the only valid decision is a `respond` message. That guarantee is what lets you safely render the choices as real UI, like buttons or a dropdown, instead of a plain-text chat box, because you know the interrupt will always be there to catch the answer.

</Tip>

Open `python/m1/m1.8_hitl_respond.py`. It defines two tools: `ask_user` and `send_email`.

### 1. Define the clarifying-question tool

```python
from langchain_core.tools import tool

@tool
def ask_user(question: str, choices: list[str]) -> str:
    """Ask the user a clarifying question with a fixed set of choices."""
    return f"[No response recorded for: {question}]"
```

The function body is a placeholder. Because `ask_user` will always be configured to only allow `respond`, the tool itself never actually runs; the human's answer is returned in its place.

### 2. Restrict each tool to its own decisions

```python
agent = create_deep_agent(
    model=model,
    tools=[ask_user, send_email],
    system_prompt=SYSTEM_PROMPT,
    interrupt_on={
        "ask_user": {"allowed_decisions": ["respond"]},
        "send_email": {"allowed_decisions": ["approve", "edit", "reject"]},
    },
    checkpointer=MemorySaver(),
)
```

`ask_user` can only be answered with `respond`. `send_email` can only be approved, edited, or rejected. Passing an `edit` decision for `ask_user`, for example, would raise an error; the framework enforces the boundary between "answer a question" and "review a side effect" for you.

### 3. Render the choices as an actual menu

```python
import questionary

while result.interrupts:
    pending = result.interrupts[0].value
    decisions = []
    for req in pending["action_requests"]:
        if req["name"] == "ask_user":
            selected = questionary.select(
                req["args"]["question"],
                choices=req["args"]["choices"],
            ).ask()
            decisions.append({"type": "respond", "message": selected})

        elif req["name"] == "send_email":
            ...  # same approve/edit/reject handling as Lab 1

    result = agent.invoke(Command(resume={"decisions": decisions}), config=config, version="v2")
```

<Tip>

`questionary` is a third-party CLI library, not part of deepagents or LangGraph. `questionary.select(question, choices=...)` just renders an arrow-key menu in the terminal and returns whichever choice the user picks. It's here purely as a stand-in UI; a real app could swap in a web form or a Slack button and the agent-side code would not change.

</Tip>

This is the payoff of restricting `ask_user` to `respond`: because the tool's args always carry a `question` and a fixed `choices` list, the CLI can hand them to `questionary.select(...)` and render a real arrow-key menu instead of a free-text `input()` prompt. Whatever the student picks becomes the `respond` message verbatim, so the model receives an unambiguous answer, not a string it has to interpret.
