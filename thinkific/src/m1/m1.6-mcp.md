## MCP: Connecting Agents to External Services

Model Context Protocol (MCP) is an open protocol that standardizes how applications provide tools and context to LLMs. Instead of each integration requiring custom code in your agent, MCP servers expose a consistent interface that any MCP-aware client can connect to.

<iframe src="images/m1.6-mcp-slides-1.html" style="width:100%;height:520px;border:none;border-radius:12px;display:block;margin:1.25rem 0 1.5rem;" title="MCP architecture walkthrough; use arrow keys or Prev/Next to navigate"></iframe>

LangChain agents connect to MCP servers using the `langchain-mcp-adapters` library. The adapters wrap each MCP tool as a standard LangChain `BaseTool`, so the agent sees it identically to any locally-defined tool. The tool result is added to the message history and the model decides what to do next, exactly as it would with a regular tool call.

### Configuration

deepagents discovers MCP servers from `.mcp.json` in your project root (or `~/.deepagents/.mcp.json` for user-level config). The project repo already includes a config pointing at the LangChain documentation server:

<iframe src="images/m1.6-mcp-slides-config.html" style="width:100%;height:380px;border:none;border-radius:12px;display:block;margin:1.25rem 0 1.5rem;" title=".mcp.json configuration diagram"></iframe>

Load those tools in Python with `get_mcp_tools`:

```python
# python/m1/m1.6_scratch_agent_mcp.py
from deepagents_code.mcp_tools import get_mcp_tools

tools, session_manager, server_infos = await get_mcp_tools(".mcp.json")
```

Pass `tools` to `create_deep_agent` as you would any other tool list. When you are done, call `await session_manager.cleanup()` to close the connections.

<details>
<summary><strong>Transport details</strong></summary>

The transport is the underlying mechanism used to communicate with an MCP server. It is abstracted away from your agent code; the agent never sees or interacts with it directly.

There are three transport types:

- **stdio** runs the server as a local subprocess on your machine. Messages pass as JSON through the subprocess's standard input and output streams. No networking required.
- **HTTP** sends a standard HTTPS request to a remote server and waits for the response. A new connection opens for each tool call and closes when the response arrives.
- **SSE** (Server-Sent Events) opens a persistent HTTP connection. The server can push events back over that connection in real time, which is useful for long-running tools or streaming results. SSE is the newest transport type in the MCP spec.

<iframe src="images/m1.6-mcp-slides-2.html" style="width:100%;height:460px;border:none;border-radius:12px;display:block;margin:1.25rem 0 1.5rem;" title="Transport types walkthrough; use arrow keys or Prev/Next to navigate"></iframe>

In `.mcp.json`, set `"type"` to `"http"` or `"sse"` for remote servers, or use `"command"` and `"args"` for stdio servers. deepagents normalizes the transport at load time; your Python code does not change.

</details>

### Authorization

Some MCP servers require the user to authenticate before the agent can call their tools.

<iframe src="images/m1.6-mcp-slides-auth.html" style="width:100%;height:460px;border:none;border-radius:12px;display:block;margin:1.25rem 0 1.5rem;" title="Authorization walkthrough; use arrow keys or Prev/Next to navigate"></iframe>

Add `"auth": "oauth"` to the server entry in `.mcp.json`:

```json
// .mcp.json
{
  "mcpServers": {
    "gmail": {
      "type": "http",
      "url": "https://gmailmcp.googleapis.com/mcp/v1",
      "auth": "oauth",
      "allowedTools": ["search_threads", "get_thread"]
    }
  }
}
```

When deepagents detects a server that requires OAuth, it pauses and opens a browser window for the user to log in. Tokens are stored in `~/.deepagents/.state/mcp-tokens/` and reused on subsequent runs.

`allowedTools` lets you restrict which tools from a server the agent can see. This is useful when a server exposes many tools but you only need a subset; for example, giving an agent read-only access to Gmail without exposing tools that can send or delete.

### Human-in-the-Loop (HITL)

Even after authorization, you may want a human to confirm before the agent reads or writes data. Pass `interrupt_on` to `create_deep_agent` with a mapping of tool names to approval settings:

<iframe src="images/m1.6-mcp-slides-hitl.html" style="width:100%;height:460px;border:none;border-radius:12px;display:block;margin:1.25rem 0 1.5rem;" title="HITL walkthrough; use arrow keys or Prev/Next to navigate"></iframe>

```python
# python/m1/m1.6_scratch_agent_mcp.py
from deepagents import create_deep_agent
from langgraph.checkpoint.memory import MemorySaver

agent = create_deep_agent(
    model=model,
    tools=tools,
    interrupt_on={
        "search_threads": True,
        "get_thread":     True,
    },
    checkpointer=MemorySaver(),
)
```

`True` pauses the agent before calling that tool and presents the pending call to the user. The user can approve, edit the inputs, reject, or respond with a message. Execution resumes once a decision is made.

<div style="margin:1.5rem 0;padding:1.25rem 1.5rem;border-radius:12px;border:1px solid #99d3ff;background-color:#e5f4ff;color:#1e4d7a;">
<strong>interrupt_on requires a checkpointer</strong>
<p style="margin:0.5rem 0 0;">When the agent pauses for human approval, it needs to save its current state so execution can resume from exactly the same point after a decision is made. A checkpointer handles that save-and-restore. <code>MemorySaver</code> keeps state in memory and is fine for development; use a persistent checkpointer in production.</p>
</div>

## Connecting to a Docs Server

This lab connects an agent to the LangChain documentation server, an HTTP MCP server maintained by LangChain, not you. When the docs update, the tools update. You write nothing; you maintain nothing.

### Step 1: Load and inspect

Open `python/m1/m1.6_scratch_agent_mcp.py`, already in the repo and connects to the LangChain docs server. Here's a walkthrough of what it does.

The first thing `main()` does is load the MCP tools and print each tool's name and description so you can see what the server is offering:

```python {12-21,25-31}
# python/m1/m1.6_scratch_agent_mcp.py
import sys
import asyncio
import json
from pathlib import Path
sys.path.insert(0, str(Path(__file__).resolve().parent.parent))

from models import model
from deepagents import create_deep_agent
from deepagents_code.mcp_tools import get_mcp_tools

MCP_CONFIG = Path(__file__).resolve().parent / ".mcp.json"

MCP_CONFIG.write_text(json.dumps({
    "mcpServers": {
        "docs-langchain": {
            "type": "http",
            "url": "https://docs.langchain.com/mcp"
        }
    }
}, indent=2))


async def main():
    tools, session_manager, server_infos = await get_mcp_tools(str(MCP_CONFIG))

    for info in server_infos:
        print(f"\n{info.name} ({info.transport}): {len(info.tools)} tool(s)")
        for t in info.tools:
            print(f"  {t.name}")
            print(f"  {t.description[:90]}")


asyncio.run(main())
```

**Example output:**

  Search across the Docs by LangChain knowledge base to find relevant...

Two tools, no code written. Each name carries the `docs-langchain_` server prefix; deepagents uses that prefix to keep tools from different servers distinct and to let `interrupt_on` and `allowedTools` target tools precisely.

### Step 2: Ask a question

The agent is already wired up in `main()`; after printing the tools, it invokes the agent and prints the response:

```python {33-41}
# python/m1/m1.6_scratch_agent_mcp.py
import sys
import asyncio
import json
from pathlib import Path
sys.path.insert(0, str(Path(__file__).resolve().parent.parent))

from models import model
from deepagents import create_deep_agent
from deepagents_code.mcp_tools import get_mcp_tools

MCP_CONFIG = Path(__file__).resolve().parent / ".mcp.json"

MCP_CONFIG.write_text(json.dumps({
    "mcpServers": {
        "docs-langchain": {
            "type": "http",
            "url": "https://docs.langchain.com/mcp"
        }
    }
}, indent=2))


async def main():
    tools, session_manager, server_infos = await get_mcp_tools(str(MCP_CONFIG))

    for info in server_infos:
        print(f"\n{info.name} ({info.transport}): {len(info.tools)} tool(s)")
        for t in info.tools:
            print(f"  {t.name}")
            print(f"  {t.description[:90]}")

    agent = create_deep_agent(model=model, tools=tools)

    result = await agent.ainvoke({
        "messages": [{"role": "user", "content": "How does LangGraph's interrupt API work, and what role does a checkpointer play?"}]
    })
    print(result["messages"][-1].content)

    if session_manager:
        await session_manager.cleanup()


asyncio.run(main())
```

```bash
cd ~/Documents/Github/lca-deepagents/python && uv run ./m1/m1.6_scratch_agent_mcp.py
```

Open LangSmith. The agent calls `docs-langchain_search_docs_by_lang_chain` one or more times, reads the results, and synthesizes an answer. The trace looks identical to a locally-defined tool; the agent cannot tell the difference.

### Step 3: Restrict with `allowedTools`

`allowedTools` is already set in `m1.6_scratch_agent_mcp.py`, restricting the server to the search tool only. Here's that config:

```python {6}
MCP_CONFIG.write_text(json.dumps({
    "mcpServers": {
        "docs-langchain": {
            "type": "http",
            "url": "https://docs.langchain.com/mcp",
            "allowedTools": ["search_docs_by_lang_chain"]
        }
    }
}, indent=2))
```

Run it and the startup output shows 1 tool instead of 2. The agent still answers the same question; it simply cannot reach the filesystem tool.

<div style="margin:1.5rem 0;padding:1.25rem 1.5rem;border-radius:12px;border:1px solid #99d3ff;background-color:#e5f4ff;color:#1e4d7a;">
<strong>allowedTools vs. disabledTools</strong>
<p style="margin:0.5rem 0 0;"><code>allowedTools</code> is an allowlist; only the named tools are kept; anything new the server adds in the future stays invisible. <code>disabledTools</code> is a denylist; only the named tools are removed; anything new the server adds is visible by default. Use <code>allowedTools</code> when you want a strict, stable subset; use <code>disabledTools</code> when you want most of the server's tools and just need to block a few.</p>

## Recap

- MCP is an open protocol that standardizes how applications expose tools and context to LLMs; any MCP-aware client connects to any MCP server without custom integration code
- `langchain-mcp-adapters` wraps each server tool as a standard `BaseTool`; the agent cannot tell MCP tools apart from locally-defined ones
- Servers are configured in `.mcp.json`; the `type` field sets the transport (`stdio` for local subprocesses, `http` for stateless remote calls, `sse` for persistent streaming connections)
- `allowedTools` restricts which tools from a server the agent can see; useful when a server exposes many tools but you only need a subset, or want to limit access to read-only operations
- `auth: "oauth"` triggers a one-time browser authorization on first run; tokens are cached in `~/.deepagents/.state/mcp-tokens/` and reused automatically
- `interrupt_on` pauses the agent before a named tool call for human approval; like HITL with custom tools, it requires a checkpointer
