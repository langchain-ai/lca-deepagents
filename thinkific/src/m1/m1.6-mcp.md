## MCP: Connecting Agents to External Services

Model Context Protocol (MCP) is an open protocol that standardizes how applications provide tools and context to LLMs. Instead of each integration requiring custom code in your agent, MCP servers expose a consistent interface that any MCP-aware client can connect to.

<iframe src="images/m1.6-mcp-slides-1.html" style="width:100%;height:520px;border:none;border-radius:12px;display:block;margin:1.25rem 0 1.5rem;" title="MCP architecture walkthrough; use arrow keys or Prev/Next to navigate"></iframe>

LangChain agents connect to MCP servers using the `langchain-mcp-adapters` library. The adapters wrap each MCP tool as a standard LangChain `BaseTool`, so the agent sees it identically to any locally-defined tool. The tool result is added to the message history and the model decides what to do next, exactly as it would with a regular tool call.

### Configuration

Connect to an MCP server by passing a server dictionary to `MultiServerMCPClient`. Each entry names the server, its transport type, and its URL:

```python
# python/m1/m1.6_agent_mcp.py
from langchain_mcp_adapters.client import MultiServerMCPClient

client = MultiServerMCPClient({
    "docs-langchain": {
        "transport": "http",
        "url": "https://docs.langchain.com/mcp",
    }
})
tools = await client.get_tools()
```

Pass `tools` to `create_deep_agent` as any other tool list.

<details>
<summary><strong>Transport details</strong></summary>

The transport is the underlying mechanism used to communicate with an MCP server. It is abstracted away from agent code; the agent never sees or interacts with it directly.

The transport types are:

- **stdio** runs the server as a local subprocess. Messages pass as JSON through the subprocess's standard input and output streams. No networking required.
- **Streamable HTTP** (`"streamable_http"`) is the current MCP standard for remote servers, introduced in spec version 2025-03-26 to replace SSE. The connection supports streaming, but by default `MultiServerMCPClient` creates a fresh session for each tool call and closes it when the response arrives. Use `client.session()` to hold a persistent connection when the server is stateful.
- **HTTP** (`"http"`) is an alias for `"streamable_http"` in `langchain-mcp-adapters`. Same transport, shorter string. Use whichever spelling you prefer.
- **SSE** (Server-Sent Events) is an older transport that opens a persistent HTTP connection for server-push events. It was deprecated in the MCP spec as of version 2025-03-26 in favor of Streamable HTTP. Don't use it for new servers; only reach for it if you're talking to a legacy server that hasn't migrated.

<iframe src="images/m1.6-mcp-slides-2.html" style="width:100%;height:422px;border:none;border-radius:12px;display:block;margin:1.25rem 0 1.5rem;" title="Transport types walkthrough; use arrow keys or Prev/Next to navigate"></iframe>

Set `transport` to `"http"` for remote servers. For stdio servers, use `"stdio"` and add `"command"` and `"args"` keys instead of `"url"`.

</details>

### Authorization

Some MCP servers require a user to sign in before the agent can call their tools. At the agent level, the idea is simple: pass an auth provider in the server dictionary alongside the transport and URL. The auth provider is responsible for getting and attaching credentials.

The advanced OAuth lab walks through a full GitHub example. Treat it as optional: the core MCP idea is that remote tools become normal LangChain tools for the agent.

```python
client = MultiServerMCPClient({
    "github": {
        "transport": "streamable_http",
        "url": MCP_URL,
        "auth": auth_provider,
    }
})
```

The optional OAuth lab walks through a complete GitHub flow, showing how to build `auth_provider` and what happens at each step.

---

## Recap

- MCP is an open protocol that standardizes how applications expose tools and context to LLMs; any MCP-aware client connects to any MCP server without custom integration code
- `langchain-mcp-adapters` wraps each server tool as a standard `BaseTool`; the agent cannot tell MCP tools apart from locally-defined ones
- `MultiServerMCPClient` takes a server dictionary with `transport` and `url`; call `get_tools()` to retrieve all tools as `BaseTool` objects
- `transport` selects the connection mechanism: `"stdio"` for local subprocesses, `"http"` (or `"streamable_http"`) for remote servers, `"sse"` for legacy servers only (deprecated in the MCP spec; don't use it for anything new)
- Some MCP servers need authorization; an auth provider handles credentials, and the optional OAuth lab shows the full flow

## Connecting to a Docs Server

This lab connects an agent to the LangChain documentation server, an HTTP MCP server maintained by LangChain.

Before running it, make sure:

- your model API key from setup is available in the Python environment;
- the course dependencies are installed, including `langchain-mcp-adapters`;
- you have internet access to `https://docs.langchain.com/mcp`.

If the agent answers without calling the MCP tool, rerun with a prompt that explicitly says "Use the LangChain docs MCP tool..." as this lab does.

### Step 1: Load and inspect

Open `python/m1/m1.6_agent_mcp.py`. The first thing `main()` does is create a `MultiServerMCPClient`, fetch all available tools, and print each tool's name and description:

```python {12-25}
# python/m1/m1.6_agent_mcp.py
import asyncio

from deepagents import create_deep_agent
from langchain_mcp_adapters.client import MultiServerMCPClient

from models import model

ALLOWED = {"search_docs_by_lang_chain"}


async def main():
    client = MultiServerMCPClient({
        "docs-langchain": {
            "transport": "http",
            "url": "https://docs.langchain.com/mcp",
        }
    })
    tools = await client.get_tools()

    print(f"\ndocs-langchain: {len(tools)} tool(s)")
    for t in tools:
        print(f"  {t.name}")
        print(f"  {t.description[:90]}")

    tools = [t for t in tools if t.name in ALLOWED]

    agent = create_deep_agent(model=model, tools=tools)

    result = await agent.ainvoke({
        "messages": [{"role": "user", "content": "Use the LangChain docs MCP tool to explain what MCP is and how LangChain uses MCP tools."}]
    })
    print(result["messages"][-1].content)


asyncio.run(main())
```

### Step 3: Filter tools

`get_tools()` returns everything the server exposes. The filter shown below narrows that to the search tool only; the agent never learns the filesystem tool exists:

```python {1,3}
ALLOWED = {"search_docs_by_lang_chain"}

tools = [t for t in tools if t.name in ALLOWED]
```

<div style="margin:1.5rem 0;padding:1.25rem 1.5rem;border-radius:12px;border:1px solid #99d3ff;background-color:#e5f4ff;color:#1e4d7a;">
<strong>Allowlist vs. denylist</strong>
<p style="margin:0.5rem 0 0;">The filter above is an allowlist. Only tools whose names are in <code>ALLOWED</code> pass through. A denylist form would be <code>[t for t in tools if t.name not in BLOCKED]</code>. Use an allowlist when you want a strict, stable subset; use a denylist when you want most tools and only need to exclude a few.</p>
<p style="margin:0.5rem 0 0;">When using the deepagents CLI, the equivalent config fields in <code>.mcp.json</code> are <code>allowedTools</code> (allowlist) and <code>disabledTools</code> (denylist).</p>

## Optional: MCP with OAuth

This section is optional and more advanced than the core MCP lab. The main idea is still simple: some MCP servers require credentials, and an auth provider gets those credentials before the agent calls the tools.

Many useful services require a user to sign in before exposing data. Increasingly, application vendors offer MCP interfaces that use OAuth: a browser-based permission flow that returns a token the agent can attach to future requests.

We will use the GitHub MCP server as our example. It requires an OAuth access token before it will respond to any request, and the endpoint used here is `https://api.githubcopilot.com/mcp/`. You may need GitHub Copilot/MCP access for your account; a normal GitHub account alone may not be enough.

This lab demonstrates static client registration, where the OAuth client is registered with the provider once before running any code. It covers two things: how to set up an agent to obtain and use a token, and what happens under the hood when it does.

Before running it, make sure:

- you have a GitHub account with access to the GitHub/Copilot MCP endpoint;
- your model API key is configured from setup;
- you can create a GitHub OAuth App;
- `GITHUB_CLIENT_ID` and `GITHUB_CLIENT_SECRET` will be added to `python/.env` after app registration.

Common failure modes:

- if the browser opens the wrong GitHub account, copy the printed URL into a browser/profile logged into the right account;
- if the callback fails, check the callback URL and port notes below;
- if auth succeeds but MCP tool calls fail, the account may not have GitHub/Copilot MCP access or the token may not have the required scope.

### Three terms before the code

- **OAuth** is the browser-based permission flow: a user signs in, approves access, and the app receives a token.
- **Authentication** means the user proves who they are by logging in.
- **Authorization** means the user grants the app specific permissions, such as `repo` or `read:user`.

## Set up your agent with OAuth

Two steps are required:

1. Register an OAuth application (a one-time setup).
   - This identifies the agent to the application. It can also be a time

2. Configure the agent with the credentials and run it.

### Register your OAuth client

Before running any of the programs below, register an OAuth App
in GitHub. This one-time manual step is what makes GitHub willing to
issue tokens to the application at all.

Follow GitHub's guide: [Creating an OAuth App](https://docs.github.com/en/apps/oauth-apps/building-oauth-apps/creating-an-oauth-app).

When filling in the registration form at
`github.com/settings/developers → New OAuth App`, use these values:

| Field | Value |
|---|---|
| Application name | `LCA-DA-Course-MCP` |
| Homepage URL | `http://localhost` |
| Authorization callback URL | `http://127.0.0.1:8765/` |

The scripts use a fixed loopback callback URL, `http://127.0.0.1:8765/`, so register that exact value for this lab. If the callback fails, make sure nothing else is using port `8765`, confirm the OAuth App callback URL matches exactly, then rerun the script.

After clicking **Register application**, immediately click **Generate a new
client secret**. GitHub shows it only once. Add both values to `.env`:

```bash
# .env
GITHUB_CLIENT_ID=your_client_id_here
GITHUB_CLIENT_SECRET=your_client_secret_here
```

`GITHUB_CLIENT_ID` is used to build the authorization URL during the OAuth
consent flow. `GITHUB_CLIENT_SECRET` is used to exchange the authorization
code for an access token.

# python/m1/gh_auth/github_agent.py
"""GitHub MCP agent — OAuth via the MCP SDK's built-in flow.

The MCP SDK's OAuthClientProvider handles PKCE, discovery, and token exchange.
GitHubTokenStorage pre-seeds the registered client credentials so the SDK skips
dynamic client registration (which GitHub does not support).

Run:  uv run python m1/gh_auth/github_agent.py
"""

from __future__ import annotations

import asyncio
import os
import threading
import webbrowser
from http.server import BaseHTTPRequestHandler, HTTPServer
from pathlib import Path
from urllib.parse import parse_qs, urlparse

from deepagents import create_deep_agent
from dotenv import load_dotenv
from langchain_mcp_adapters.client import MultiServerMCPClient
from langchain_mcp_adapters.tools import load_mcp_tools
import httpx
from mcp.client.auth import OAuthClientProvider
from mcp.shared.auth import OAuthClientInformationFull, OAuthClientMetadata, OAuthToken

from models import model

ENV_PATH = Path(__file__).resolve().parent.parent.parent / ".env"
load_dotenv(dotenv_path=ENV_PATH, override=True)

TOKEN_FILE = Path(__file__).resolve().parent.parent.parent / ".m1_github_token"

MCP_URL = "https://api.githubcopilot.com/mcp/"
REDIRECT_URI = "http://127.0.0.1:8765/"
SCOPE = "repo read:user"


class GitHubOAuthClientProvider(OAuthClientProvider):
    def __init__(self, *args, requested_scope: str, **kwargs) -> None:
        super().__init__(*args, **kwargs)
        self._requested_scope = requested_scope

    async def _perform_authorization_code_grant(self) -> tuple[str, str]:
        self.context.client_metadata.scope = self._requested_scope
        return await super()._perform_authorization_code_grant()

    async def _exchange_token_authorization_code(
        self, auth_code: str, code_verifier: str, *, token_data: dict | None = {}
    ) -> httpx.Request:
        req = await super()._exchange_token_authorization_code(
            auth_code, code_verifier, token_data=token_data
        )
        req.headers["Accept"] = "application/json"
        return req


class GitHubTokenStorage:
    def __init__(self, client_id: str, client_secret: str) -> None:
        self._client_id = client_id
        self._client_secret = client_secret

    async def get_tokens(self) -> OAuthToken | None:
        if TOKEN_FILE.exists():
            raw = TOKEN_FILE.read_text().strip()
            if raw:
                try:
                    return OAuthToken.model_validate_json(raw)
                except Exception:
                    return OAuthToken(access_token=raw, token_type="Bearer")
        return None

    async def set_tokens(self, tokens: OAuthToken) -> None:
        TOKEN_FILE.write_text(tokens.model_dump_json())

    async def get_client_info(self) -> OAuthClientInformationFull:
        return OAuthClientInformationFull(
            client_id=self._client_id,
            client_secret=self._client_secret,
            redirect_uris=[REDIRECT_URI],
            token_endpoint_auth_method="client_secret_post",
        )

    async def set_client_info(self, info: OAuthClientInformationFull) -> None:
        pass


async def open_browser(url: str) -> None:
    print(f"Opening browser for GitHub login ({SCOPE})...")
    print(f"If it doesn't open, paste this URL:\n{url}\n")
    webbrowser.open(url)


async def wait_for_callback() -> tuple[str, str | None]:
    result: dict[str, str] = {}
    done = threading.Event()

    class Handler(BaseHTTPRequestHandler):
        def do_GET(self):  # noqa: N802
            params = parse_qs(urlparse(self.path).query)
            result["code"] = params.get("code", [""])[0]
            result["state"] = params.get("state", [""])[0]
            self.send_response(200)
            self.send_header("Content-Type", "text/html")
            self.end_headers()
            self.wfile.write(b"<h3>Authorized. You can close this tab.</h3>")
            done.set()

        def log_message(self, *a): pass

    port = int(urlparse(REDIRECT_URI).port or 80)
    server = HTTPServer(("127.0.0.1", port), Handler)
    threading.Thread(target=server.serve_forever, daemon=True).start()

    if not done.wait(timeout=300):
        server.shutdown()
        raise SystemExit("Timed out waiting for OAuth callback.")
    server.shutdown()
    return result["code"], result.get("state")


async def main() -> None:
    client_id = os.environ.get("GITHUB_CLIENT_ID")
    client_secret = os.environ.get("GITHUB_CLIENT_SECRET")
    if not client_id or not client_secret:
        raise SystemExit("Set GITHUB_CLIENT_ID and GITHUB_CLIENT_SECRET in .env.")

    auth = GitHubOAuthClientProvider(
        server_url=MCP_URL,
        client_metadata=OAuthClientMetadata(
            redirect_uris=[REDIRECT_URI],
            scope=SCOPE,
            token_endpoint_auth_method="client_secret_post",
        ),
        storage=GitHubTokenStorage(client_id, client_secret),
        redirect_handler=open_browser,
        callback_handler=wait_for_callback,
        requested_scope=SCOPE,
    )

    client = MultiServerMCPClient({
        "github": {
            "transport": "streamable_http",
            "url": MCP_URL,
            "auth": auth,
        }
    })

    async with client.session("github") as session:
        tools = await load_mcp_tools(session)
        print(f"github: {len(tools)} tool(s) available")

        agent = create_deep_agent(model=model, tools=tools)
        print("Running agent...\n")
        result = await agent.ainvoke({
            "messages": [{"role": "user", "content": (
                "What is my GitHub username and what repositories do I have?"
            )}]
        })
        print("\n" + result["messages"][-1].content)


if __name__ == "__main__":
    asyncio.run(main())

`OAuthClientProvider` is from the MCP Python SDK and implements `httpx.Auth`, the interface httpx uses to inject authentication into every request automatically. It handles the full OAuth flow: it follows the `WWW-Authenticate` header to discover the authorization server, generates PKCE parameters, calls `redirect_handler` to open the browser, awaits `callback_handler` to receive the authorization code, exchanges the code for a token, and caches it via `storage`.

`GitHubTokenStorage` pre-seeds the registered `client_id` and `client_secret` so the SDK skips dynamic client registration, which GitHub does not support. `GitHubOAuthClientProvider` is a small subclass with two GitHub-specific fixes: it restores the requested scope (the SDK otherwise uses GitHub's full advertised scope list), and adds `Accept: application/json` to the token exchange request (GitHub returns URL-encoded form data by default).

On first run (no cached token) the program opens the browser for consent; on subsequent runs it reads the cached token and skips straight to the agent:

## Under the hood

The OAuth flow involves five actors:

- **the user**, who approves access in the browser;
- **the client** (`myagent`), the component that wants to call tools;
- the **browser**, which carries the user to GitHub to log in and returns with the
  authorization code;
- **the authorization server** (GitHub), which logs the user in and issues tokens;
- **the resource server** (the GitHub MCP server), which holds the protected
  tools.

    participant M as GitHub MCP<br/>Server

    M-->>A: 401 + resource metadata → endpoints

    B-->>A: authorization code

    O-->>A: access_token
    A->>M: Retry with Bearer token
    M-->>A: Tool result

The `WWW-Authenticate` header in the `401` points to the protected-resource
metadata URL. Fetching it returns the `authorization_servers` and
`scopes_supported` fields shown in the output. GitHub publishes no RFC 8414
discovery document at that server URL, so `authorization_endpoint` and
`token_endpoint` are derived from the issuer rather than looked up. Those two
endpoints and the scopes are everything Step 2 needs.

### Step 2 — Get the user's consent (`step2.py`)

This step generates Proof Key for Code Exchange (PKCE, pronounced 'pixie') values, opens the browser for login and consent, and

Two things happened, and each is worth separating.

**PKCE protected the code in transit.** Before opening the browser, the script
generated two values:

- a `code_verifier`, a high-entropy secret kept in memory and never sent
  through the browser;
- a `code_challenge`, the SHA-256 hash of the verifier, sent in the
  authorization URL (`code_challenge_method=S256`).

GitHub staples the challenge to the code it issues. In Step 3, redeeming the
code requires presenting the original verifier; GitHub re-hashes it and checks
it matches. The payoff: the authorization code travels through the browser and
a redirect URL, both places where it can leak, but a stolen code is useless
without the verifier, which never left the script.

**The browser round-trip produced an authorization code.** After the user consented,
GitHub redirected back to a one-shot listener on `127.0.0.1` carrying the code.
The script also sent a random `state` value and checks that the redirect
returns the same one, which prevents a forged callback from injecting a code.

> **Note on the browser.** The script calls `webbrowser.open()`, which uses
> the operating system's default browser. If that opens the wrong account,
> copy the URL printed after "If it doesn't open" and paste it into a browser
> window logged into the correct GitHub account.

The captured string `df2407422664...` is an opaque, short-lived,
single-use authorization code. It has no decodable structure; it is a handle
that only GitHub's token endpoint can redeem, and only when presented alongside
the `code_verifier` from PKCE. Step 3 performs that exchange.

The authorization code signifies that a real user approved the request. The
token itself is never sent through the browser; redirect URLs end up in
browser history and server logs, so the design keeps the token out of that
path entirely. The code crosses the browser boundary safely because it is
worthless without the verifier and client secret, which never left the script.

The token exchange is a direct server-to-server POST with no browser involved.
The script sends the authorization code, the `code_verifier` from PKCE, the
`redirect_uri` (must match what Step 2 used), and the `client_secret`. GitHub
re-hashes the `code_verifier` against the `code_challenge` it stored when it
issued the code, and if they match, it returns an access token.

The access token `gho_TGboFbwQ...` is the credential that authorizes the
agent to act. The `gho_` prefix identifies it as a GitHub OAuth token; it is
40 characters of opaque data, not a JWT. With it, the same requests that
returned `401` in Step 1 now succeed.

Step 2 saved the authorization code and `code_verifier` to a temp file
(`.m1_github_state.json`) so Step 3 could read them as a separate program.
Step 3 deletes that file immediately after the exchange. The code is
single-use and is now spent.

## Recap

- A protected MCP server gates every request behind a `401`. That `401` is
  self-describing: its `WWW-Authenticate` header points to resource metadata
  naming the **authorization server** and **scopes** (RFC 9728).
- Authorization servers vary in what they publish. GitHub does not publish
  an RFC 8414 discovery document; when discovery is absent, endpoints are
  derived from the issuer URL.
- OAuth proceeds in two halves: get the user's **authorization code** (Step 2),
  then exchange it for an **access token** (Step 3).
- **PKCE** binds the code to a secret the client never transmits, so a leaked
  code is worthless on its own.
- Authorization codes and GitHub access tokens are **opaque bearer handles**;
  the server validates them by lookup, not by reading their contents.
- An agent can manage its own OAuth end-to-end (probe, consent, cache, connect)
  without any manual steps between the phases.

    if (!panel) return;
    var unrendered = Array.from(panel.querySelectorAll('.mermaid')).filter(function (div) {

  function show(p) {
    tabs.forEach(function (t) {
      var on = t.getAttribute('data-p') === p;
      t.classList.toggle('active', on);
      t.setAttribute('aria-selected', on ? 'true' : 'false');
    });
    document.querySelectorAll('.lt-panel').forEach(function (panel) {
      panel.classList.toggle('active', panel.id === 'p-' + p);
    });
    activatePanel(p);
  }

  tabs.forEach(function (t) {
    t.addEventListener('click', function () { show(t.getAttribute('data-p')); });
  });
