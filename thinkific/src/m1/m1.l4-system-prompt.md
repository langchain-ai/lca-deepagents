## The System Prompt

The system prompt **guides the model and is part of every model call**; it is
re-sent on every turn, ahead of the conversation. `create_deep_agent` assembles
it from up to **three parts**:

<table>
<thead>
<tr><th>Part</th><th>Where it comes from</th></tr>
</thead>
<tbody>
<tr><td><strong>USER</strong></td><td>The <code>system_prompt</code> argument <em>you</em> pass to <code>create_deep_agent</code>. Optional.</td></tr>
<tr><td><strong>BASE</strong></td><td>The SDK's built-in <code>BASE_AGENT_PROMPT</code> (behavior guidance). Always present, unless a harness profile replaces it.</td></tr>
<tr><td><strong>SUFFIX</strong></td><td>Model-specific tuning from a <strong>harness profile</strong> (<code>system_prompt_suffix</code>). Optional.</td></tr>
</tbody>
</table>

Putting that together:

- `create_deep_agent` takes those three parts and **joins them with blank lines**
  into a single prompt.
- That combined string is the `system_prompt` it hands to the model on every call.
- The diagram below walks through how the pieces stack up, one step at a time.

> **What is middleware?** Middleware are pluggable components that hook into the
> agent loop at defined points. A middleware can run logic `before_model` and
> `after_model` (and `before_agent` / `after_agent`), wrap each model call to
> rewrite the outgoing request, intercept tool calls, register extra tools, and
> read or extend the agent's state; some, like human-in-the-loop, can even pause
> (interrupt) the run to wait for input. You never call middleware directly,
> `create_deep_agent` installs the required ones for you.

On top of those three assembled parts, **mandatory middleware append more
sections to the same `system` message at request time**:

- the filesystem tool instructions,
- a `write_todos` section, and
- the `task` (subagent) section.

So what you read in **LangSmith** is BASE + SUFFIX *plus* those middleware
additions, all in one `system` message. The required deep-agent middleware do
this through a `wrap_model_call` hook, which also registers their tools.

These are the two ways to set the system prompt your agent ships with:

<summary>Extension: Harness Profiles (Under The Hood) [click to expand]</summary>

The **SUFFIX** is model-specific tuning, and it comes from a **harness profile**:
a small bundle of defaults the SDK attaches automatically based on the model id.
A profile sets a `system_prompt_suffix` (the SUFFIX, appended after BASE), and it
can also set a `base_system_prompt` that replaces BASE outright (the CUSTOM
variant) or drop optional middleware.

The SDK ships built-in profiles for several models (they live in
`deepagents/profiles/harness/`):

<table>
<thead>
<tr><th>Model</th><th>What its profile adds as the SUFFIX</th></tr>
</thead>
<tbody>
<tr><td><code>anthropic:claude-haiku-4-5</code> <em>(our default)</em></td><td>parallel tool calls, investigate-before-answering, reflect on tool results</td></tr>
<tr><td><code>anthropic:claude-sonnet-4-6</code></td><td>parallel tool calls, grounded (non-speculative) answers, reflection</td></tr>
<tr><td><code>anthropic:claude-opus-4-7</code></td><td>universal Anthropic guidance plus Opus-4.7-specific steering</td></tr>
<tr><td><code>openai:gpt-5.x-codex</code></td><td>autonomous-engineer demeanor, bias to action, parallel tools</td></tr>
<tr><td>anything with no registered profile (e.g. <code>gpt-4.1-mini</code>)</td><td>nothing; no SUFFIX at all</td></tr>
</tbody>
</table>

That's why, back in Lab 1, swapping the model changed the tail of the system
prompt: each model brings its own SUFFIX, or none. You normally never touch
these; the SDK tunes each model to behave well out of the box. (If you ever need
to, `register_harness_profile(...)` lets you add your own.)

> One nuance the trace makes clear: the SUFFIX is appended to the **BASE prompt
> text**, not to the end of the whole message. It's the last thing in the
> assembled `system_prompt` string; the middleware sections (`write_todos`,
> filesystem, `task`) are then added as separate blocks *after* that.

</details>

## Prompt Assembly

`create_deep_agent` joins its three parts in a **fixed order**, separated by
blank lines (this is the top half of the assembly slideshow earlier in the lesson):

```text
USER   →   (BASE or CUSTOM)   →   SUFFIX
 ↑                ↑                  ↑
your       SDK base, or a        harness-profile
prompt   profile's replacement       tuning
```

Then the **mandatory middleware append their own sections** onto that same
`system` message at request time, *after* the SUFFIX:

```text
... SUFFIX   →   write_todos   →   filesystem   →   task
                 (todo mw)        (filesystem mw)   (subagent mw)
```

The takeaway isn't the exact ordering. Models don't reliably weight earlier text
over later, so don't over-think where each piece sits. What matters is that all
three parts (USER, BASE, SUFFIX) land in one assembled `system_prompt`, and the
middleware sections (`write_todos`, filesystem, `task`) are then appended
separately by the SDK. Write clear instructions; let the SDK handle the layout.

This part is conceptual; there's no separate lab for it. To see it for real,
open any run in **LangSmith** and read the single assembled `system` message top
to bottom. Because the workshop default is `claude-haiku-4-5`, its harness
profile **does** define a SUFFIX, so you'll actually see one: a block of
`<use_parallel_tool_calls>`, `<investigate_before_answering>`, and
`<tool_result_reflection>` tags, sitting after the BASE behavior guidance.
(A model with no registered profile shows no SUFFIX.)

## Lab 2: The System Prompt

**The discovery:** `system_prompt` is free-form text; use it to set a persona, lay down constraints, or define the agent's domain. The SDK slots it in as the **USER** segment at the front of the assembled prompt, leaving the BASE scaffolding untouched.

Open `python/m1/scratch_agent.py` and make two changes: add a `SYSTEM_PROMPT` variable, then pass it to `create_deep_agent`:

:::python
```python {1,2,5}
SYSTEM_PROMPT = "You are a salty pirate captain..."  # ← your text here

from models import model
from deepagents import create_deep_agent
agent = create_deep_agent(model=model, system_prompt=SYSTEM_PROMPT)
```
:::

<details>
<summary>Persona ideas [click to expand]</summary>
<br>

Pick one of these or write your own:

```python
# Pirate
SYSTEM_PROMPT = (
    "You are a salty pirate captain with decades at sea. Talk in 'arr' and nautical slang. "
    "Call the user 'matey' and frame your answers as if charting a course."
)

# Cowboy
SYSTEM_PROMPT = (
    "You are a drawling cowboy from the Old West. Speak only in cowboy slang, partner. "
    "Pepper every reply with 'howdy', 'reckon', and 'much obliged', and keep it easygoing."
)

# Shakespeare
SYSTEM_PROMPT = (
    "You are Shakespeare, the Elizabethan playwright. Reply only in Early Modern English. "
    "Use 'thee' and 'thou', and close with a short rhyming couplet when you can."
)
```

Each one changes the agent's voice without touching anything else. That single
string is your USER segment.

</details>

If you used the pirate persona, the reply should come back with 'arr' and nautical slang. That's the USER segment taking effect.

### Check the trace in LangSmith

Open the run in **LangSmith**, click the LLM call, and read the first `system`
message. Your `system_prompt` shows up verbatim as the **USER** segment, and the
rest of the message is the same scaffolding you saw in Lab 1. Switch personas,
re-run, and you'll see only your line change while everything else stays put, proof
that this single string is all you touched to redirect the agent's behavior.
