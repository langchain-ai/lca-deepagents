## What is middleware?

> **What is middleware?** Middleware are pluggable components that hook into the
> agent loop at defined points. A middleware can run logic `before_model` and
> `after_model` (and `before_agent` / `after_agent`), wrap each model call to
> rewrite the outgoing request, intercept tool calls, register extra tools, and
> read or extend the agent's state; some, like human-in-the-loop, can even pause
> (interrupt) the run to wait for input. You never call middleware directly,
> `create_deep_agent` installs the required ones for you.

## Middleware in prompt assembly

On top of those three assembled parts, **mandatory middleware append more
sections to the same `system` message at request time**:

- the filesystem tool instructions,
- a `write_todos` section, and
- the `task` (subagent) section.

So what you read in **LangSmith** is BASE + SUFFIX *plus* those middleware
additions, all in one `system` message. The required deep-agent middleware do
this through a `wrap_model_call` hook, which also registers their tools.

## Prompt Assembly section

`create_deep_agent` joins its three parts in a **fixed order**, separated by
blank lines (this is the top half of the assembly slideshow earlier in the lesson):

```text
USER   →   (BASE or CUSTOM)   →   SUFFIX
 ↑                ↑                  ↑
your       SDK base, or a        harness-profile
prompt   profile's replacement       tuning
```

Then the **mandatory middleware append their own sections** onto that same
`system` message at request time, *after* the SUFFIX:

```text
... SUFFIX   →   write_todos   →   filesystem   →   task
                 (todo mw)        (filesystem mw)   (subagent mw)
```

The takeaway isn't the exact ordering. Models don't reliably weight earlier text
over later, so don't over-think where each piece sits. What matters is that all
three parts (USER, BASE, SUFFIX) land in one assembled `system_prompt`, and the
middleware sections (`write_todos`, filesystem, `task`) are then appended
separately by the SDK. Write clear instructions; let the SDK handle the layout.

This part is conceptual; there's no separate lab for it. To see it for real,
open any run in **LangSmith** and read the single assembled `system` message top
to bottom. Because the workshop default is `claude-haiku-4-5`, its harness
profile **does** define a SUFFIX, so you'll actually see one: a block of
`<use_parallel_tool_calls>`, `<investigate_before_answering>`, and
`<tool_result_reflection>` tags, sitting after the BASE behavior guidance.
(A model with no registered profile shows no SUFFIX.)
