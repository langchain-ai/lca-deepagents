## Tools: Extending the Agent

<Video src="https://share.descript.com/view/E2a0vp8gM5Y" />

Tools are functions you give an agent so it can act: look things up, run queries, call APIs, write files. The agent decides when to call them and what arguments to pass; you define what they do.

Every deep agent ships with a built-in set of tools, loaded automatically by the SDK. You already saw these appear in the LangSmith trace in the previous lesson, as `write_todos` and `Filesystem Tools` inside the system message:

<table>
<thead>
<tr><th>Tool(s)</th><th>What it does</th></tr>
</thead>
<tbody>
<tr><td><code>ls</code>, <code>read_file</code>, <code>write_file</code>, <code>edit_file</code></td><td>Read and write files in the agent's working directory</td></tr>
<tr><td><code>glob</code>, <code>grep</code></td><td>Search files by name pattern or content</td></tr>
<tr><td><code>execute</code></td><td>Run shell commands (requires a sandbox backend)</td></tr>
<tr><td><code>task</code></td><td>Delegate work to a sub-agent with an isolated context</td></tr>
<tr><td><code>write_todos</code></td><td>Manage a running to-do list for multi-step planning</td></tr>
</tbody>
</table>

These are always present. The `tools` parameter on `create_deep_agent` is **additive**: whatever you pass is merged with the built-in suite, not substituted for it.

```python
agent = create_deep_agent(
    model=model,
    tools=[my_tool_a, my_tool_b],   # merged with the built-ins above
)
```

### Defining a tool with `@tool`

<Video src="https://share.descript.com/view/oOMDbVt36V9" />

Tools are part of the LangChain library. The `@tool` decorator from `langchain_core.tools` is the standard way to define one: write a regular Python function and the decorator turns it into something the model can call. LangChain also ships ready-made tools for common tasks (for example, `TavilySearchResults` for web search), so you don't have to write everything from scratch.

```python
from langchain_core.tools import tool

@tool
def get_stock_price(ticker: str) -> str:
    """Return the latest closing price for a stock ticker symbol."""
    ...  # your implementation
```

<Image src="images/m1.5-tool-anatomy.svg" width="800px" alt="Diagram showing the three parts of a @tool definition: the decorator (@tool) registers it as a LangChain tool, the function name and type annotations form the name and schema, and the docstring is the description the model reads to decide when to call it." />

The description is the most important of the three. Write it as a clear, specific sentence: it is the only signal the model has for deciding when to reach for this tool versus another.

The return value should be a string (or anything JSON-serializable). It comes back to the model as the tool result and is visible in the conversation.

### Human-in-the-Loop

> **Human-in-the-Loop (HITL)** is a pattern where agent execution pauses at a critical step and waits for a human to approve, modify, or cancel before continuing. Deep agents support this via the `interrupt_on` argument to `create_deep_agent`, which intercepts named tool calls before they execute and presents them for review. It is important because agents can take irreversible actions: writing to a database, sending a message, deleting a file. HITL gives you a checkpoint before those actions happen.

## Recap

- The `tools=` parameter is additive; your tools merge with the built-in suite (`write_todos`, `read_file`, `write_file`, `edit_file`, `glob`, `grep`, `task`) ; nothing is replaced
- The `@tool` decorator turns a Python function into a callable tool; the docstring is the description the model reads to decide when to use it
- The LLM never executes tools directly; it outputs an `AIMessage` with a `tool_calls` list; the Tool Node reads that and runs the function
- Human-in-the-Loop uses `interrupt_on` on `create_deep_agent` to intercept named tool calls before they execute; `Command(resume={"decisions": [...]})` restores execution after a human decision
- HITL requires a checkpointer (`MemorySaver` for development) so the graph can save state on pause and resume from exactly the same point

  function retreat() { current <= 0 ? showAll() : goTo(current - 1); }

  showAll();
  prevBtn.addEventListener('click', retreat);
  nextBtn.addEventListener('click', advance);
  dots.forEach(function(d, i) { d.addEventListener('click', function() { goTo(i); }); });
  stage.addEventListener('click', advance);
  stage.addEventListener('keydown', function(e) {
    if (e.key === 'ArrowRight' || e.key === ' ') { e.preventDefault(); advance(); }
    if (e.key === 'ArrowLeft') { e.preventDefault(); retreat(); }
  });
  var tx = 0;
  stage.addEventListener('touchstart', function(e) { tx = e.touches[0].clientX; }, { passive: true });
  stage.addEventListener('touchend', function(e) {
    var dx = e.changedTouches[0].clientX - tx; if (Math.abs(dx) > 40) dx < 0 ? advance() : retreat();
  });
})();

A checkpointer saves the agent's state at each step so execution can pause and later resume from exactly the same point. `MemorySaver` is the simplest checkpointer; it keeps that state in memory. It is required for HITL: without it, the graph has no way to pause mid-run and pick up again after a human decision.

### 2. System prompt

Two tools means the agent needs to know which to reach for:

```python
SYSTEM_PROMPT = """You are a SQL analyst with access to the Chinook music store database.

Rules:
- Use read_sql for all SELECT queries.
- Use write_sql for INSERT, UPDATE, DELETE, and ALTER operations.
- Think step-by-step. Read first, then write.
- If a tool returns an error, revise the SQL and try again.
- Show your SQL in your final answer.
"""
```

---

### 3. Define `read_sql` and `write_sql`

`read_sql` executes and returns immediately. `write_sql` is a plain tool with no HITL logic — the interrupt is handled by the `interrupt_on` argument to `create_deep_agent`, which intercepts the call before the Tool Node runs it.

```python
@tool
def read_sql(query: str) -> str:
    """Run a read-only SELECT query against the Chinook music store database."""
    try:
        return db.run(query)
    except Exception as e:
        return f"Error: {e}"

@tool
def write_sql(query: str) -> str:
    """Execute a write operation (INSERT, UPDATE, DELETE, ALTER) against the Chinook database.
    Requires human approval before executing."""
    try:
        return db.run(query)
    except Exception as e:
        return f"Error: {e}"
```

---

### 4. Wire it up

Pass both tools, the checkpointer, and `interrupt_on`. The `while` loop handles the approval prompt and resumes the graph with your decision:

```python
checkpointer = MemorySaver()

agent = create_deep_agent(
    model=model,
    tools=[read_sql, write_sql],
    system_prompt=SYSTEM_PROMPT,
    checkpointer=checkpointer,
    interrupt_on={"write_sql": True},
)

config = {"configurable": {"thread_id": "lab3"}}

result = agent.invoke(
    {"messages": [{"role": "user", "content": "What genres are in the database? Then add a new genre called 'Synthwave'."}]},
    config=config,
)

while result.get("__interrupt__"):
    pending = result["__interrupt__"][0].value
    for req in pending["action_requests"]:
        print(f"\nApproval required for {req['name']}:")
        print(f"  {req['args']}")
    approval = input("\nApprove? (yes/no): ").strip().lower()
    decision = "approve" if approval in ("yes", "y") else "reject"
    decisions = [{"type": decision} for _ in pending["action_requests"]]
    result = agent.invoke(Command(resume={"decisions": decisions}), config=config)

print(result["messages"][-1].content)
```

### Take it further: build your own tools

The same `@tool` pattern works for anything Python can do. Describe what you want to your coding agent and let it write the implementation, then drop it into `create_deep_agent(tools=[...])`.

Three ideas to get started:

- **REST API**: fetch live data from any JSON API: weather forecasts, GitHub issues, stock prices. One `@tool` that calls `requests.get(...)` and returns the result is all it takes.
- **File reader**: give the agent access to local CSVs, JSON exports, or PDFs. A tool that reads and returns file contents lets it analyse data you already have on disk.
- **Action tool**: tools don't have to be read-only. Post to Slack, create a calendar event, write a row to a spreadsheet. The agent can act, not just retrieve.
