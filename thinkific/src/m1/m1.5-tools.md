## Tools: Extending the Agent

Tools are functions you give an agent so it can act: look things up, run queries, call APIs, write files. You define what they do; the agent decides when to call them and what arguments to pass.

Every deep agent ships with a built-in set of tools, loaded automatically by the SDK. You already saw these appear in the LangSmith trace in the previous lesson, as `write_todos` and `Filesystem Tools` on the model call:

<table>
<thead>
<tr><th>Tool(s)</th><th>What it does</th></tr>
</thead>
<tbody>
<tr><td><code>ls</code>, <code>read_file</code>, <code>write_file</code>, <code>edit_file</code></td><td>Read and write files in the agent's working directory</td></tr>
<tr><td><code>glob</code>, <code>grep</code></td><td>Search files by name pattern or content</td></tr>
<tr><td><code>execute</code></td><td>Run shell commands (requires a sandbox backend)</td></tr>
<tr><td><code>task</code></td><td>Delegate work to a sub-agent with an isolated context</td></tr>
<tr><td><code>write_todos</code></td><td>Manage a running to-do list for multi-step planning</td></tr>
</tbody>
</table>

These are always present. The `tools` parameter on `create_deep_agent` is **additive**: whatever you pass is merged with the built-in suite, not substituted for it.

```python
agent = create_deep_agent(
    model=model,
    tools=[my_tool_a, my_tool_b],   # merged with the built-ins above
)
```

Step through the sequence to see how each part fits together. The model asks for a tool call, and **Tools**, the built-in runner inside the agent, executes the Python function and returns the result.

### Defining a tool with `@tool`

<details style="border:2.5px solid #000;border-radius:6px;background:#fff;margin:1rem 0;"><summary style="padding:10px 16px;cursor:pointer;font-weight:500;font-family:'IBM Plex Mono',monospace;font-size:14px;">Video Walkthrough</summary><div style="padding:12px 16px 16px;"><div class="video-container" style="max-width:750px;"><div class="video-wrapper"><iframe src="https://share.descript.com/embed/oOMDbVt36V9" frameborder="0" allow="autoplay; fullscreen; encrypted-media; picture-in-picture" allowfullscreen></iframe></div></div></div></details>

Tools are part of the LangChain library. The `@tool` decorator from `langchain_core.tools` is the standard way to define one: write a regular Python function and the decorator turns it into something the model can call. LangChain also ships ready-made tools for common tasks (for example, `TavilySearchResults` for web search), so you don't have to write everything from scratch.

```python
from langchain_core.tools import tool

@tool
def get_stock_price(ticker: str) -> str:
    """Return the latest closing price for a stock ticker symbol."""
    ...  # your implementation
```

<Image src="images/m1.5-tool-anatomy.svg" width="800px" alt="Diagram showing the three parts of a @tool definition: the decorator (@tool) registers it as a LangChain tool, the function name and type annotations form the name and schema, and the docstring is the description the model reads to decide when to call it." />

The description is the most important natural-language signal. Write it as a clear, specific sentence: the model uses it alongside the tool name and argument schema to decide when to reach for this tool versus another.

The return value should be a string (or anything JSON-serializable). It comes back to the model as the tool result and is visible in the conversation.

### A note on action tools

Tools can do more than retrieve information. They can also take actions: write to a database, send a message, create a ticket, or delete a file. Those tools need safety controls. In Lesson 8, we'll cover Human-in-the-Loop approval so a person can review sensitive tool calls before they execute.

## Recap

- The `tools=` parameter is additive; your tools merge with the built-in suite (`write_todos`, `ls`, `read_file`, `write_file`, `edit_file`, `glob`, `grep`, `execute`, `task`); nothing is replaced
- The `@tool` decorator turns a Python function into a callable tool; the docstring is the most important natural-language description the model reads to decide when to use it
- The LLM never executes tools directly; it outputs an `AIMessage` with a `tool_calls` list; Tools reads that and runs the function
- Tools can retrieve information or take actions; Lesson 8 covers how to add Human-in-the-Loop approval for sensitive action tools

### 1. Imports and database connection

Here's how the file sets up its imports and database connection:

```python
import warnings
warnings.filterwarnings("ignore", category=DeprecationWarning)

from pathlib import Path

from deepagents import create_deep_agent
from langchain_community.utilities import SQLDatabase
from langchain_core.tools import tool

from models import model

DB_PATH = Path(__file__).parent / "chinook.db"
db = SQLDatabase.from_uri(f"sqlite:///{DB_PATH}")
```

---

### 2. System prompt

The system prompt tells the agent what the tool is for and asks it to show the SQL it used:

```python
SYSTEM_PROMPT = """You are a SQL analyst with access to the Chinook music store database.

Rules:
- Use read_sql for SELECT queries.
- Do not modify the database.
- If a tool returns an error, revise the SQL and try again.
- Show your SQL in your final answer.
"""
```

---

### 3. Define `read_sql`

`read_sql` is a regular Python function. The `@tool` decorator turns it into a tool the agent can call.

```python
@tool
def read_sql(query: str) -> str:
    """Run a read-only SELECT query against the Chinook music store database."""
    try:
        return str(db.run(query))
    except Exception as e:
        return f"Error: {e}"
```

---

### 4. Wire it up

Pass the tool into `create_deep_agent`. The custom `read_sql` tool is merged with the built-in Deep Agents tools.

```python
agent = create_deep_agent(
    model=model,
    name="SQL Agent",
    tools=[read_sql],
    system_prompt=SYSTEM_PROMPT,
)

result = agent.invoke(
    {"messages": [{"role": "user", "content": "Which five genres have the most tracks?"}]}
)

print(result["messages"][-1].content)
```

---

### Run it

```bash
cd python
uv run ./m1/m1.5_scratch_agent_tools.py
```

The agent should call `read_sql`, inspect the query result, and answer with the top genres plus the SQL it used.

<div class="callout">
<strong>What is happening</strong><br>
The model does not query SQLite directly. It emits a <code>read_sql</code> tool call with a SQL string as the argument. Tools runs your Python function, wraps the database result in a ToolMessage, and sends that result back to the model. The final answer is generated from that tool result.

### Take it further: build your own tools

The same `@tool` pattern works for anything Python can do. Describe what you want to your coding agent and let it write the implementation, then drop it into `create_deep_agent(tools=[...])`.

Three ideas to get started:

- **REST API**: fetch live data from any JSON API: weather forecasts, GitHub issues, stock prices. One `@tool` that calls `requests.get(...)` and returns the result is all it takes.
- **Action tool**: tools don't have to be read-only. Post to Slack, create a calendar event, write a row to a spreadsheet. The agent can act, not just retrieve.
