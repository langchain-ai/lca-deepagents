When a conversation runs long enough, older messages push the total token count toward the model's context limit. Deep Agents handles this automatically with **SummarizationMiddleware**; it watches the context fill up and compacts old messages into a summary before the model ever sees them.

---

## How the context window fills up

Think of the context window like a thermometer. Every new message adds tokens and the level rises. When it crosses **85%** of the model's limit, the middleware fires.

## What the summary contains

The middleware makes a separate LLM call to generate the summary. It uses a structured prompt that asks the model to extract:

- **Session intent**: what the user is trying to accomplish
- **Key facts & decisions**: important context from the conversation
- **Artifacts**: files or resources created or modified
- **Next steps**: what remains to do

The resulting text is injected as a single `HumanMessage` at the top of the preserved recent messages. The model picks up the conversation as if it had been there the whole time.

<details>
<summary>Summarization details (for LangGraph users)</summary>
<br>

**`state["messages"]` never shrinks.** Every message ever sent is retained in state. The list only grows.

**The effective list is transient.** Before each model call the middleware assembles the list the model actually sees; it is never stored back in state:

```python
effective_list = [S_n] + state["messages"][cutoff_index:]
#                 ↑ oldest (index 0)          ↑ recent messages
```

`S_n` is the summary from the last compaction event. It sits at index 0, the oldest position, acting as a stand-in for everything that was dropped.

**`state["_summarization_event"]`** is the only state that changes at compaction time:

```python
{
    "cutoff_index":    int,          # ignore state["messages"][:cutoff_index]
    "summary_message": HumanMessage, # S_n: the summary text
}
```

**Rolling compaction (n → n+1).** When the effective list fills up again, the middleware assembles it the same way. `S_n` is at index 0 of that list, so it gets included in what is sent to the summarizer, absorbed into the new `S_n+1`. The `cutoff_index` advances further into `state["messages"]`, permanently skipping the messages that are now covered by `S_n+1`.

```
After compaction n:
  state["messages"]   = [m0  m1  m2  m3  m4 | m5  m6  m7]
                                              ↑ cutoff_index = 5
  effective list      = [S_n  m5  m6  m7]

After more turns, compaction n+1 fires:
  state["messages"]   = [m0  m1  m2  m3  m4  m5  m6  m7  m8  m9  m10  m11]
  effective list      = [S_n  m5  m6  m7  m8  m9  m10  m11]   ← over threshold
  summarize           = [S_n  m5  m6  m7  m8]  →  S_n+1
  cutoff_index        = 8 (skips m0–m7 forever)
  effective list      = [S_n+1  m9  m10  m11]
```

`m0–m4` and `m5–m8` remain in `state["messages"]` but are never read again.

</details>

## Seeing it in action

The default trigger is 85% of the model's real context window: 170,000 tokens for Claude Haiku 4.5. To observe it in a demo we set `model.profile["max_input_tokens"]` to a small value so the threshold is hit after a few turns:

<Tabs>
<Tab title="Python">

```python
# python/m2/m2.4_MessageHistory.py
MODEL = "anthropic:claude-haiku-4-5"
model = init_chat_model(MODEL)

# Lower the reported context window so summarization fires at ~1700 tokens
# instead of the real 170,000-token threshold.
model.profile = {**model.profile, "max_input_tokens": 700}

agent = create_deep_agent(
    model=model,
    checkpointer=MemorySaver(),
    system_prompt="You are a helpful assistant. Keep every response to one sentence.",
)
```

</Tab>
</Tabs>

<Tip>

**Passing `model` not `MODEL`.** Notice `create_deep_agent(model=model, ...)` receives the model *object*, not the string `"anthropic:claude-haiku-4-5"`. Passing the string would re-initialize the model internally and ignore the `profile` override.

</Tip>

When summarization fires, a private `_summarization_event` key appears in the agent's state. The `state["messages"]` list itself is never modified; the summarization happens only on what gets *sent* to the model each turn.

## Recap

- **SummarizationMiddleware** intercepts every LLM request via `wrap_model_call`; it runs before the model sees any messages
- When total tokens exceed 85% of `max_input_tokens`, old messages are partitioned at the **keep cutoff** and replaced with a summary
- The summary is generated by a separate LLM call using the same model as the agent
- `state["messages"]` is never mutated; summarization is tracked in a private `_summarization_event` field
- Passing `model.profile["max_input_tokens"]` overrides the threshold, making it practical to demo
