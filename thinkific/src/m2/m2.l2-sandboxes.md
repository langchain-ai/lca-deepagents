## Sandboxes

`FilesystemBackend` gives the agent file tools: read, write, edit, search. A sandbox backend adds one more: `execute`, which runs shell commands. The distinction matters because executing arbitrary code is a different category of action from reading and writing files.

### The `execute` tool

When the agent has a sandbox backend, it gains access to `execute(command, timeout)`. The agent can use this to run scripts it has written, invoke CLI tools, compile code, or chain shell commands. The output is returned as a tool result and fed back into the next LLM call.

A typical pattern: the agent writes a script with `write_file`, then runs it with `execute`, reads the output, and revises if needed. You can see this sequence in LangSmith as a chain of tool calls.

### Why isolation matters

Without isolation, `execute` runs directly on your machine with your permissions. That is acceptable for local development but not for production agents or any situation where the agent might run untrusted or generated code. A sandbox limits what the process can access: filesystem scope, network access, resource limits, and process lifetime.

### Available backends

The SDK ships two concrete sandbox implementations:

| Backend | Where commands run | Use case |
|---|---|---|
| `LocalShellBackend` | Your local machine, unrestricted | Local development and testing |
| `LangSmithSandbox` | Isolated LangSmith-managed environment | Production or shared deployments |

Both extend `FilesystemBackend`, so they include all the file tools from the previous lesson in addition to `execute`.

`LocalShellBackend` is the simplest option: no API key or extra setup required. Commands run as your user with no network or filesystem restrictions. Use it to explore what the agent does, not to run untrusted code.

`LangSmithSandbox` runs commands in an isolated environment managed by LangSmith. It requires a LangSmith API key and project configuration, and is the right choice once you move beyond local experimentation.

### Configuring a sandbox backend

```python
from deepagents.backends import LocalShellBackend

agent = create_deep_agent(
    model=model,
    backend=LocalShellBackend(root_dir="workspace/"),
)
```

Swapping in `LangSmithSandbox` requires no other code changes:

```python
from deepagents.backends import LangSmithSandbox

agent = create_deep_agent(
    model=model,
    backend=LangSmithSandbox(root_dir="workspace/"),
)
```

The agent sees the same tools either way. The backend determines only where execution happens.

## Lab 6: Sandbox Agent

You will build an agent that writes a Python script to disk, executes it, and reports the output. This demonstrates the write-then-execute pattern and shows the full tool sequence in LangSmith.

This lab uses `LocalShellBackend`: commands run directly on your machine. Do not ask the agent to run untrusted or destructive commands.

---

### 1. Create `sandbox_agent.py`

Create `python/m2/sandbox_agent.py`:

```python
import sys
from pathlib import Path
from deepagents.backends import LocalShellBackend
from deepagents import create_deep_agent
sys.path.insert(0, str(Path(__file__).resolve().parent.parent))
from models import model

WORKSPACE = Path(__file__).parent / "workspace"
WORKSPACE.mkdir(exist_ok=True)

agent = create_deep_agent(
    model=model,
    backend=LocalShellBackend(root_dir=str(WORKSPACE)),
    system_prompt=(
        "You are a coding assistant. When asked to run code, write the script to a file "
        "first, then execute it. Show the output in your final answer."
    ),
)

result = agent.invoke({"messages": [{"role": "user", "content": "Write a Python script that prints the first 15 Fibonacci numbers, save it to fib.py, and run it."}]})
print(result["messages"][-1].content)
```

Before running, try a different task:

- `"Write a Python script that counts word frequencies in a short paragraph of your choice, save it to wordcount.py, and run it."`
- `"Write a Python script that generates a multiplication table for numbers 1 to 5, save it to times-table.py, and run it."`

---

### Run it

```bash
cd ~/Documents/Github/lca-deepagents/python && uv run ./m2/sandbox_agent.py
```

---

### What to check

Open `python/m2/workspace/` and confirm the script file was created. Then open **LangSmith** and look at the tool call sequence:

1. `write_file`: the agent writes the script
2. `execute`: the agent runs it
3. The tool result carries the stdout output back to the LLM
4. The LLM includes the output in its final answer

If the script had an error, you will see the agent call `edit_file` to fix it and `execute` again. This self-correction loop is one of the key behaviours a sandbox enables.

<div class="callout">
<strong>LocalShellBackend vs LangSmithSandbox</strong><br>
To use <code>LangSmithSandbox</code> instead, replace the import and the <code>backend=</code> argument. Make sure <code>LANGSMITH_API_KEY</code> is set in your <code>.env</code>. No other code changes are needed; the agent sees the same tools either way.
