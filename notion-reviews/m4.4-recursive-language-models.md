# Recursive Language Models (RLMs)

## From one subagent to many

Earlier in this module (m4.2), delegation meant handing **one task** to **one subagent** and getting one result back: a `genre-researcher` for Rock, another call for Metal, a third for Latin. You wrote the `task` calls yourself, one per genre, because there were only four genres.

Now say the job is 10,000 sales transcripts, same task on every one: summarize it. There's no longer a short list the model can enumerate in its head, and someone still has to track which ones are done.

- Leave that to the model and you've put the exact thing that breaks agents at scale, tracking state across a shrinking context window, back in its head instead of off of it.

> In practice, a model asked to take on all 10,000 transcripts itself wouldn't just cleanly fail. Depending on the model, it would more likely:
>
> - Refuse outright, saying it can't process that many in one pass.
> - Try to chunk the work into rough ranges across subagent calls, e.g. transcripts 0-1,000, then 1,000-2,000, and so on, hoping it remembers where it left off.
> - Try to fire off everything in batches of roughly 20 calls at a time, since that's close to the practical ceiling on how many tool calls a model can issue in parallel.
>
> None of these guarantee every transcript actually gets touched; each still depends on the model, not code, getting the bookkeeping right.

Here's that same difference written as code, since that's really what's changing:

```python
# the model enumerates each item itself, and has to remember its own place
task(description="summarize transcript 1 of 10,000")
task(description="summarize transcript 2 of 10,000")
# ...however many calls it takes, nothing guarantees it reaches transcript 10,000...
```

```python
# written once, run by code: the loop is the guarantee, not the model's memory
for transcript in transcripts:
    task(description=f"summarize {transcript}")
```

`task()` itself didn't change; it's the same subagent call either way. What changed is *where the decision to keep going lives*:

- **First version:** in the model, however many calls it takes.
- **Second version:** in a loop the interpreter runs once.

This is where **RLMs**, recursive language models, come in: subagents dispatched by code, not by the model.

That loop doesn't have to be a literal `for`: it could be a map over a list, a parallel fan-out, or a branching pipeline. The only thing that matters is that it's a fixed shape written in code, not something the model has to reconstruct from memory on every turn.

> **Diagram: "Same subagent primitive, two different scales"**
>
> Two side-by-side panels.
>
> - **Left — small, fixed N (shown in Module 4):** a main agent fans out to four subagents at once, in a single step. Note: *"the model enumerates the 4 items itself; 4 is easy to hold in context."*
> - **Right — large / open-ended N (RLM):** a main agent writes a loop once (`for item in items: task(item)`), and the loop fans out to many subagents at once — however many there are. Note: *"code enumerates the list; N could be 10,000, and the model never tracks it."*

The subagent on the right is doing the exact same job as the one on the left; even the fan-out pattern is the same. What's different is *who enumerated the list*: at 4 items, the model could hold that itself. At 10,000, only code reliably can. That's the whole idea in one picture.

## RLM = recursive language model

An **RLM** is a model that can call itself. The term comes from Alex Zhang and MIT CSAIL researchers, who proposed Recursive Language Models to fight **context rot**, the same context management issue you saw in Module 3 (Summarization and Context Offloading).

Instead of cramming everything into one model's context window, an RLM runs code in an interpreter that dispatches subagents and recurses over pieces of the input, treating the prompt as a variable it can inspect and manipulate programmatically, using primitives you'd recognize from big data: **grep, partition, map, reduce**.

Their core insight: instead of one massive model call, split the context into two model calls, then combine their results in a third.

> **Diagram (interactive, 5 steps): "Split, call, combine"**
>
> 1. **Split, call** — A `context` box splits into `call A (half 1)` and `call B (half 2)`.
> 2. **Combine in a third call** — Both results feed into a third call, `call C: combine(A, B)`.
> 3. **Recursing on a piece** — If a half is still too big to handle in one call, that call recurses the same way: `call A (still large)` splits again into `A1` and `A2`. Call C can't run yet; it's waiting on A.
> 4. **A1 and A2 hit the base case** — `A1` and `A2` are small enough to answer directly now, so they combine into `call A3: combine(A1, A2)`. Call C is still waiting on that result.
> 5. **The loop closes** — Call A3's result feeds back up into call C. The loop closes: recursion bottoms out, it doesn't split forever.

- **Orchestration lives in code.** The loop, the branching, the "did I get to all of them" logic, is a script, not a chain of model turns.
- **Context lives in variables or files**, not in the model's own transcript. A running total is a Python variable, not something the model has to keep re-stating to itself.
- **Divide and conquer.** Split the work, process the pieces (possibly recursively), combine.

## Why RLMs

Agents have trouble doing work reliably **at scale**, for two structural reasons:

- **Context windows are finite.** There's a hard ceiling on how much a single model call can hold.
- **Summarization is lossy.** Compressing history to fit is the usual fix, and it's the fix you already met in m3.1, but compression throws information away.

RLMs support two things that turn-by-turn delegation doesn't guarantee:

- **Deterministic coverage.** A `for batch in batches` loop touches every batch. A model deciding for itself, turn by turn, whether it's covered everything reliably fails at scale.
  - On a long-context benchmark called OOLONG (below), a plain agent at 128k tokens didn't just answer wrong; it often gave up outright, reporting it couldn't compute the result.
- **Bespoke orchestration.** The pipeline can branch, run in parallel, or run sequentially, shaped however the task needs, instead of being constrained to one tool call per model turn.

> **OOLONG benchmark**: a long-context reasoning benchmark where the answer depends on examining nearly every row of the input, not just retrieving an isolated fact, making it a useful stress test for context rot. This run classifies headlines into four categories, then asks counting, user-filtered, and temporal questions over nearly all of them (0–1 scale):
>
> | Context length | Without interpreter | With interpreter |
> |---|---|---|
> | 64k tokens | 0.58 | 0.67 |
> | 128k tokens | 0.44 | 0.79 |
>
> The interpreter-enabled agent was slower and cost more in output tokens, and this was an untuned smoke test, not a finished benchmark. Treat the gap as directional evidence, not a guarantee.

## How Deep Agents implements this: dynamic subagents

You already have both ingredients this needs: **subagents** (m4.1–m4.2) and a **code interpreter** (m2.4). LangChain's implementation, **dynamic subagents**, wires them together: instead of the model calling `task` turn by turn, it writes a script that the interpreter executes, and that script dispatches subagents programmatically.

The canonical example is one subagent per page of a 300-page document:

```javascript
const results = await Promise.all(pages.map(page =>
  task({
    description: `Summarize page ${page.number}`,
    subagentType: "summarizer",
    responseSchema: summarySchema, // -> { summary: string }
  })
));
```

> **Diagram (collapsible): "300-page fan-out example"**
>
> Two side-by-side panels.
>
> - **Left — small, fixed N (shown in Module 4):** a main agent fans out to four subagents at once, in a single step. Note: *"the model enumerates a short list itself."*
> - **Right — this example: 300-page document:** a main agent runs `Promise.all(pages.map(task))` once, and that call fans out to one `summarizer` subagent per page (`page 1`, `page 2`, ... `page 300`). Note: *"code enumerates the pages instead."*
>
> A dashed box below the right panel, connected by a dashed line, reads *"+ responseSchema (optional)"* and notes that `responseSchema` is optional and parses each result into structured JSON (here, `{ summary }`).

`Promise.all` over `pages` *is* the `for` loop on the right. The `summarizer` subagent is a subagent the way `genre-researcher` was, just dispatched by a loop in code instead of by the model turn by turn.

> `responseSchema` is optional, and available on any `task()` call whether dispatched by the model or by a loop, but it pairs especially well here: with hundreds of subagents returning results, structured output is what makes those results usable in code instead of just more text for the model to read.

> **Naming note.** Deep Agents' dynamic subagents dispatch from code, but each one is still a full agent with its own tools and state, not a bare recursive model call. The LangChain team's more precise term for this is **recursive agents (RA)**; "RLM" is the popular shorthand used throughout this lesson.

A key differentiator over a fixed orchestrator + fixed subagent setup: the orchestrator and its subagents can run on **any model or mix of models**, for example a frontier orchestrator paired with cheaper open-weight subagents for cost efficiency, or the reverse for deep research work where the subagents need to be strong.

## Patterns

The model picks a strategy from the shape of the task, and it emerges from how it writes interpreter code, not from any configuration you set. A few common shapes:

**Classify and act:** items are classified first, then each is routed to a specialist based on its category.

Good for triaging support tickets, error logs, or user feedback, any batch of items that need different handling depending on their type.

> **Diagram: "Classify and act"**
>
> A task flows into a classifier, which routes it by category (bug, feature, or question) to one of three agents: Agent A (bug-fixer), Agent B (feature-analyst), or Agent C (support-agent).

```typescript
const SPECIALIST = { bug: "bug-fixer", feature: "feature-analyst", question: "support-agent" };

const handled = await Promise.all(
  tickets.map((ticket) =>
    task({
      description: `Handle this ${ticket.category}:\n${ticket.text}`,
      subagentType: SPECIALIST[ticket.category],
    }),
  ),
);
```

**Fan-out and synthesize:** the same kind of work is dispatched across many items in parallel, then the results are combined.

Good for code review across a directory, analyzing a batch of documents, or running the same check across many services.

> **Diagram: "Fan-out and synthesize"**
>
> A task with files from a glob pattern fans out in parallel to three agents, each a reviewer on one file. Their results merge into a single result: an issues list.

```typescript
const files = (await tools.glob({ pattern: "src/routes/**/*.ts" })).split("\n").filter(Boolean);

const reviews = await Promise.all(
  files.map((file) =>
    task({
      description: `Review ${file} for authentication issues. Cite line numbers.`,
      subagentType: "reviewer",
      responseSchema: issuesSchema, // -> { issues: [{ file, line, severity }] }
    }),
  ),
);

const issues = reviews.flatMap((r) => r.issues);
```

**Adversarial verification:** a first pass produces findings, a second pass sends each one to an independent verifier, and only agreed-upon findings survive.

Good for security audits or compliance checks, anywhere a false positive is costly enough to be worth a second, independent pass.

> **Diagram: "Adversarial verification"**
>
> A task goes to Agent A, a reviewer, which audits and produces findings. Each finding fans out in parallel to an independent verifier: Agent B, C, and D. Agents B and C confirm their finding and those results flow into a final result of confirmed findings; Agent D refutes its finding, shown crossed out and dropped.

```typescript
const { findings } = await task({
  description: "Audit the payments module for vulnerabilities.",
  subagentType: "reviewer",
  responseSchema: findingsSchema, // -> { findings: [{ id, file, line, description }] }
});

const verdicts = await Promise.all(
  findings.map((f) =>
    task({
      description: `Verify ${f.file}:${f.line} (${f.description}). Confirm or refute.`,
      subagentType: "verifier",
      responseSchema: verdictSchema, // -> { confirmed: boolean }
    }),
  ),
);

const confirmed = findings.filter((_, i) => verdicts[i]?.confirmed);
```

**Tournament:** candidates are compared head-to-head by a judge, with winners advancing through elimination rounds.

Good for picking the best of several candidates when you can't score them independently, only compare two at a time, style rewrites, competing implementations, or anywhere the criteria are subjective.

> **Diagram: "Tournament"**
>
> Four candidate agents are compared pairwise by two judges in round one. Each round's winner advances to a final judge, which picks the overall winner as the result.

```typescript
const rewrites = await Promise.all(
  Array.from({ length: 4 }, (_, i) =>
    task({
      description: `Rewrite processOrder for clarity. Attempt ${i + 1}.`,
      subagentType: "writer",
    }),
  ),
);

let bracket = rewrites;
while (bracket.length > 1) {
  const next = [];
  for (let i = 0; i < bracket.length; i += 2) {
    const { winner } = await task({
      description: `Compare these two rewrites and pick the better one:\n\nA:\n${bracket[i]}\n\nB:\n${bracket[i + 1]}`,
      subagentType: "judge",
      responseSchema: winnerSchema, // -> { winner: "A" | "B" }
    });
    next.push(winner === "A" ? bracket[i] : bracket[i + 1]);
  }
  bracket = next;
}

const best = bracket[0];
```

## When to reach for this vs. a plain subagent team

> **Reach for dynamic subagents / RLMs when:**
> - The number of units of work is large, fixed, or only known at runtime (pages, rows, files, transcripts)
> - You need a **guarantee** that every unit gets covered, not a best-effort pass
> - The orchestration itself is non-trivial: branching, parallel fan-out, or a pipeline shape the model would otherwise have to reconstruct every turn
>
> **Stick with an ordinary subagent team (m4.2) when:**
> - There's a small, fixed number of specialized roles (a researcher per genre, a reviewer, an analyst)
> - You're delegating *by role*, not fanning out over a large, uniform collection of items

> **Governance note.** `task()` dispatches from inside an already-running `eval` call, so a per-dispatch `interrupt_on` approval gate doesn't apply to it the way it would to a model-initiated tool call. If you need a human in the loop before orchestration runs, gate the `eval` tool itself, not the individual subagent dispatches inside it.

## Recap

- An **RLM** (recursive language model) is a model that can call itself; the term comes from research aimed at fighting **context rot**.
- The subagent primitive from m4 doesn't change. What moves is *who writes the loop*: code, not the model's own turn-by-turn reasoning.
- Orchestration lives in code; context lives in variables or files, not the model's transcript.
- The payoff is **deterministic coverage** and **bespoke orchestration**, both hard to get from a model deciding turn by turn.
- Deep Agents implements this as **dynamic subagents**: subagents (m4) dispatched from a script running in a **code interpreter** (m2.4), each one a full agent with its own tools and state (see the naming note above on **recursive agents (RA)**).
- Reach for it when the number of work units is large or open-ended and you need guaranteed coverage; stick with a plain subagent team for a small, fixed set of specialist roles.

## Try it: a workflow-triggered dynamic subagent

> This lesson doesn't yet have a dedicated dataset or a recorded LangSmith trace the way the m4.2 newsletter lab does. What follows is a minimal, self-contained example you can run against your own files to see the pattern in action; a fuller lab with its own data is a good candidate for a follow-up pass on this module.

**1. Install the interpreter extra:**

```bash
pip install -U "deepagents[quickjs]"
```

**2. Build an agent with the interpreter enabled, and a `summarizer` subagent for it to dispatch:**

```python
import os

from deepagents import create_deep_agent
from langchain_quickjs import CodeInterpreterMiddleware
from langchain_openai import ChatOpenAI

summarizer = {
    "name": "summarizer",
    "description": "Summarize one file or one chunk of text in 3-5 sentences.",
    "system_prompt": "You are a careful summarizer. Return only the summary.",
}

model = ChatOpenAI(
    model="nvidia/nemotron-3-ultra-550b-a55b:free",
    base_url="https://openrouter.ai/api/v1",
    api_key=os.environ["OPENROUTER_API_KEY"],
)

agent = create_deep_agent(
    model=model,
    middleware=[CodeInterpreterMiddleware()],
    subagents=[summarizer],
)
```

**3. Ask for a workflow over a real, multi-file target**, for example every file in a small directory of notes or source files:

```python
result = await agent.ainvoke({
    "messages": [{
        "role": "user",
        "content": (
            "Run a workflow that summarizes every markdown file under ./notes/ "
            "and lists the three files with the most action items."
        )
    }]
})
```

**4. Compare against the turn-by-turn version.** Ask the same agent, without the word "workflow," to "summarize each file in ./notes/ one at a time." On a small folder both approaches will look similar. As the folder grows past a handful of files, watch for where the turn-by-turn version starts skipping files or losing track of the running list. That's context rot showing up in miniature, and it's exactly what the `for`-loop-in-code version is built to avoid.

## Check your understanding (quiz)

**1. What is the core problem RLMs are designed to address?**
- Slow API response times: individual model calls taking too long to return, which RLMs address by batching many requests into a single round trip
- ✅ Context rot: agent performance degrading as context accumulates, and the lossiness of summarization as a fix
- The rising dollar cost of running multiple models in parallel, which RLMs are designed to reduce by routing most work to a single cheaper model
- A lack of available subagent types for specialized tasks, which RLMs solve by letting a single agent dynamically generate new subagent types on demand

*Explanation: RLMs were proposed to combat context rot, the degradation in agent performance as context accumulates, since summarization as a fix is lossy and can cause drift on long-running aggregation tasks.*

**2. In the RLM pattern, what changes compared to the turn-by-turn subagent delegation from earlier in this module (m4.1-m4.2), and what stays the same?**
- The subagent primitive itself changes into a new recursive-only construct; the dispatch loop still stays entirely in the model's head, decided turn by turn as before
- Nothing changes at all between the two approaches; RLMs are a completely unrelated pattern that doesn't build on the subagent delegation covered earlier in this module
- ✅ The subagent primitive stays the same; what changes is that the dispatch loop moves into code the interpreter runs, instead of the model deciding turn by turn
- Subagents are replaced entirely by a single recursive model call that inlines all the work itself, with no separate `task()` dispatch or code-driven loop involved

*Explanation: A dynamic subagent does the same kind of focused task a subagent from m4 does. What moves is who writes the loop: a script the code interpreter runs dispatches many subagents, rather than the model deciding turn by turn whether it has covered everything.*

**3. What two benefits does moving orchestration into code give you over letting the model manage a loop across turns?**
- Lower token usage and faster responses in every case, since moving the loop into code always means fewer total model calls than turn-by-turn delegation
- ✅ Deterministic coverage (every unit is guaranteed to be touched) and bespoke orchestration (branching, parallel, or sequential, shaped by the task)
- The ability to use only a single model for every subagent, which removes the need to pick different subagent types for different kinds of work
- Automatic summarization of all subagent outputs into one combined report, so the main agent never has to read or process individual results itself

*Explanation: A for-loop in code guarantees every unit is processed, something a model reasoning turn by turn struggles with at scale. Code-based orchestration can also be branching, parallel, or sequential in whatever shape the task needs, rather than one tool call per model turn.*

**4. What best describes a subagent dispatched by `task()` inside an orchestration script?**
- A bare recursive call directly to the underlying model, with no tools, state, or agent loop of its own to speak of
- ✅ A full agent with its own tools and state, sometimes called a recursive agent (RA)
- A cached response computed once and then reused across every subsequent dispatch, so repeated `task()` calls skip running an agent entirely
- A single shared agent instance that gets reused across every `task()` call in the script, rather than a fresh agent being created each time

*Explanation: Each subagent dispatched by `task()` is a full agent, with its own tools and state, not a bare recursive model call. That's part of why the LangChain team also calls this pattern recursive agents (RA).*

**5. What two components does an agent need before it can use dynamic subagents / RLM-style dispatch?**
- A vector database and an embedding model, so it can retrieve semantically similar past dispatches before deciding how to orchestrate new ones
- ✅ Subagents and a code interpreter (e.g. `CodeInterpreterMiddleware`)
- A checkpointer and a thread ID, so it can persist and resume the orchestration script's state across separate conversation turns
- A sandbox environment and a Gmail-style mock MCP server, so the orchestration script has a safe tool to test its dispatch logic against

*Explanation: Dynamic subagents combine the two pieces from earlier modules: subagents (m4) to do the focused work, and a code interpreter (m2.4), such as `CodeInterpreterMiddleware`, to run the script that dispatches them.*
